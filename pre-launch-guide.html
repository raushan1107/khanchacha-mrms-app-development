Before You Go Live — The Field Guide

Four things every team must handle before a real app reaches real users. Migrate real data from Google Sheets, protect production with environments, export your Flutter app for every platform, and check security before trusting AI-generated code.

The real situation — Khan Chacha's data lives in Google Sheets

Khan Chacha currently manages maintenance requests through Google Forms. Every form submission lands as a row in a Google Spreadsheet. This is real operational data — outlet names, equipment types, request dates, status updates, who raised each request. Before the MRMS app can be useful to actual staff, this data needs to move to PostgreSQL.

From Raushan — Why this matters

You have built a beautiful app with dummy data. The managers and staff at Khan Chacha will open it and ask: "Where are our 400 requests from the last two years?" If the answer is "they're still in the spreadsheet," the app is not production-ready. Data migration is not a technical afterthought — it is a requirement, and it is the difference between a demo and a real product.

The schema gap — why you cannot just import the CSV directly

The Google Sheets export will have columns that look roughly like this:

Typical Google Forms / Sheets columns for a maintenance log

Sheets columnWhat it actually contains
TimestampDate and time of form submission — format: "7/10/2026 14:32:05"
Your NameStaff member's name — free text
Outlet / BranchOutlet name — free text, may have spelling variations
Equipment TypeDropdown: "AC", "Electrical", "Kitchen" — may not match our ENUM exactly
Equipment NameFree text: "AC Unit - Meeting Room", "Grill #2"
Description of IssueFree text, multi-line
Current StatusMay say "Done", "Pending", "In Progress" — not matching our ENUM
Assigned ToFree text — not a user ID

Your PostgreSQL schema has: id, product_name, product_id, category (ENUM), status (ENUM: Open/Pending/Closed), outlet_id (FK), raised_by (FK to users), request_date (timestamp). The two structures do not match. You cannot paste the CSV into pgAdmin and expect it to work.

What "schema gap" means in practice

  • Missing IDs — Sheets has no product_id or id columns. You must generate these.
  • ENUM mismatch — Sheets says "Done", your DB requires "Closed". Every value must be mapped.
  • FK references instead of names — Sheets says "Khan Chacha Connaught Place". Your DB needs outlet_id = 'OUT-01'.
  • Date format — Sheets: "7/10/2026 14:32:05". PostgreSQL wants: "2026-07-10T14:32:05".
  • No user FK — Sheets has a name, not a user ID from your users table.

The correct approach — schema first, migration second

1
Export the Sheets data as CSV
File → Download → Comma-separated values. Save it as sheets_export.csv.
2
Audit the CSV with Claude Code
Let Claude read the CSV and list every column, every unique value in dropdown columns, date formats, and row count. This is your gap analysis.
3

If the Sheets has 12 columns of data you need, your DB schema must accommodate all 12. Add columns, adjust ENUMs, or add free-text columns where needed. Update db/schema.sql first.

Insert all outlets and users into the DB before inserting maintenance requests. Requests have foreign keys to both — they cannot exist without their parent rows.

Step 1 — Export from Google Sheets

  1. Open the Google Spreadsheet containing the maintenance request data.
  2. Go to File → Download → Comma-separated values (.csv).
  3. Save the file as sheets_export.csv in your project folder (e.g., inside khan_chacha_api/db/).
  4. Open the CSV in VS Code or Excel and count the rows — note this number. You'll verify the same count in PostgreSQL after migration.

Step 2 — Audit with Claude Code

Claude Code — Audit Prompt (run in khan_chacha_api/)
Read the file db/sheets_export.csv and give me a complete audit: 1. List every column name exactly as it appears in the header row 2. For each column, show: data type observed, sample values (first 3 rows), and any blank cells count 3. For any column that looks like a dropdown/category, list every unique value that appears 4. Show the date format used in any date/time columns 5. Count total rows (excluding header) 6. Identify any columns that clearly map to our DB schema: - outlets table: id, name, address - users table: id, name, role, outlet_id - maintenance_requests table: id, product_name, product_id, category, status, request_date, description, outlet_id, raised_by Do not write any migration code yet. Just report the audit findings.

Step 3 — Generate the migration with Claude Code

After the audit, you will know the mapping. Now ask Claude to write the migration script. The prompt must be specific about every transformation required.

Claude Code — Migration Script Prompt
Read db/sheets_export.csv and generate a SQL migration script: db/migrate_from_sheets.sql The script must: 1. Use a transaction: BEGIN; ... COMMIT; so it can be rolled back if anything fails 2. Map columns from CSV to DB as follows (fill in after your audit): - [Sheets column "Outlet Name"] → look up outlet_id from outlets table by name - [Sheets column "Staff Name"] → look up raised_by from users table by name - [Sheets column "Equipment Type"] → category field, map values: "AC" or "Air Conditioning" → 'AC' "Electrical" or "Electric" → 'Electrical' "Kitchen" or "Kitchen Equipment" → 'Kitchen' - [Sheets column "Status"] → status field, map values: "Done" or "Completed" or "Resolved" → 'Closed' "Pending" or "In Progress" or "Open" → 'Open' anything else → 'Open' - [Sheets column "Timestamp"] → request_date, convert to ISO 8601 format 3. Generate unique IDs: 'REQ-' || LPAD(ROW_NUMBER()::text, 4, '0') (e.g. REQ-0001, REQ-0002, ...) 4. Set product_id = 'EQ-' || LPAD(ROW_NUMBER()::text, 3, '0') for each row 5. Skip any row where outlet or staff name cannot be matched — log skipped rows as a SQL comment 6. After the INSERT, run verification queries: SELECT COUNT(*) FROM maintenance_requests; -- should match CSV row count SELECT status, COUNT(*) FROM maintenance_requests GROUP BY status; -- distribution check Generate the complete db/migrate_from_sheets.sql file.

Step 4 — Verify before touching production

# Run on dev database ONLY first
psql -U kc_app -d khan_chacha_dev -f db/migrate_from_sheets.sql

# Check row count matches your CSV count
psql -U kc_app -d khan_chacha_dev -c "SELECT COUNT(*) FROM maintenance_requests;"

# Spot-check: pick 5 random rows and verify they match the spreadsheet
psql -U kc_app -d khan_chacha_dev -c "SELECT * FROM maintenance_requests ORDER BY RANDOM() LIMIT 5;"

# Check for orphaned records (rows with no matching outlet)
psql -U kc_app -d khan_chacha_dev -c "
  SELECT r.id FROM maintenance_requests r
  LEFT JOIN outlets o ON r.outlet_id = o.id
  WHERE o.id IS NULL;"

✅ Only proceed to production when

  • Row count in PostgreSQL matches row count in the CSV
  • 5 spot-checked records match the original spreadsheet exactly
  • Zero orphaned records (no missing FK references)
  • The Flutter app shows the real data correctly on the dev environment

Why environments exist

Every real software project runs in at least three parallel copies of itself. One copy is where you develop and experiment. One copy is where you test and validate. One copy is what actual users see. These are called environments — and keeping them separate is one of the most important habits in professional software development.

Khan Chacha analogy

Khan Chacha tests new menu items in one outlet before rolling them out across all 20 outlets. They don't add an untested dish to every outlet simultaneously. Software environments work the same way. You develop and test a new feature in isolated copies of the system. Only when it's validated does it go to production — where real users and real data live.

🚫 What happens without environments

Without environments, every change you make during development goes directly to production. A bug in a migration script wipes production data. A half-finished feature breaks the app for actual users. A typo in an SQL UPDATE corrupts thousands of real records. This is not hypothetical — it has happened to real companies with real consequences.

Managing databases per environment

Each environment gets its own database. They use the same schema — but they hold different data and must never share connections.

# Create the three databases on your local PostgreSQL
psql -U postgres -c "CREATE DATABASE khan_chacha_dev;"
psql -U postgres -c "CREATE DATABASE khan_chacha_test;"
# Production is on AWS RDS — created via the AWS console

# Apply the same schema to all three
psql -U kc_app -d khan_chacha_dev  -f db/schema.sql
psql -U kc_app -d khan_chacha_test -f db/schema.sql

# Seed dev with dummy data
psql -U kc_app -d khan_chacha_dev  -f db/seed.sql

# Test gets a copy of real data (after migration is verified)
# Production gets the real migration — last, after everything is validated

Managing .env files for each environment

Each environment needs its own set of credentials. The pattern is simple: one .env file per environment, never committed to Git.

.env.development

DB_HOST=localhost
DB_PORT=5432
DB_NAME=khan_chacha_dev
DB_USER=kc_app
DB_PASSWORD=kc_secure_2026
PORT=3000

.env.test

DB_HOST=staging.rds.aws.com
DB_PORT=5432
DB_NAME=khan_chacha_test
DB_USER=kc_app_test
DB_PASSWORD=[different]
PORT=3000

.env.production

DB_HOST=prod.rds.aws.com
DB_PORT=5432
DB_NAME=khan_chacha
DB_USER=kc_app_prod
DB_PASSWORD=[strong unique]
PORT=3000

In server.js, load the right file based on the environment:

// server.js — first line
const envFile = process.env.NODE_ENV === 'production'
  ? '.env.production'
  : process.env.NODE_ENV === 'test'
  ? '.env.test'
  : '.env.development';

require('dotenv').config({ path: envFile });

Add all three env files to .gitignore. Commit a .env.example with blank values so the team knows what variables are needed — but never the actual values.

The golden rules of environment management

RuleWhy
Never run migration scripts directly on productionAlways run on dev → test → verify → production
Never use the production database for development testingA bug in your WHERE clause could wipe real data
Take a backup before every production migrationpg_dump the production DB before running anything that changes data
Every schema change must be backward-compatible until old code is removedAdding a column is safe. Removing or renaming one is not — do it in two steps.
Production credentials belong to no one personUse AWS IAM roles or a secrets manager, not a password in someone's email
If something breaks in production — roll back, do not fix forward under pressureRushed fixes under stress cause more damage than the original bug
From Raushan — A real story

In one of my enterprise engagements, a developer ran a test migration script that used DELETE FROM maintenance_requests — intending to clear the test table — but had connected to the production database by mistake. Six months of data, gone. The backup was three days old. They recovered 95% of it, but 5% was permanently lost. This is why environment discipline is not optional. It is the difference between an incident and a disaster.

Understanding what "deploying" a Flutter app actually means

A Node.js API is deployed to a server — it runs on that server and waits for incoming requests. A Flutter app works completely differently. Flutter compiles your Dart code into a native binary for the target platform. That binary is distributed to devices as an installable file. The server never runs Flutter — it runs on the user's device.

❌ What Flutter apps do NOT do

  • Run on a web server
  • Need a server to "serve" them to users
  • Get updated by restarting a server
  • Live at a URL that users visit in a browser (unless using Flutter Web separately)

✅ What Flutter apps DO

  • Compile into a platform-native binary file
  • Get installed on a device like any other app
  • Run entirely on the user's device
  • Call the API over the internet for data
  • Update via Play Store / App Store / direct file replacement

Before you build — change the BASE_URL

When you build a release APK, the API server is no longer on localhost. It is on AWS. The only change needed in the Flutter code before building:

// lib/services/api_service.dart — change this one line
// From:
static const String _baseUrl = 'http://localhost:3000/api';

// To:
static const String _baseUrl = 'https://api.khanchacha.com/api';

Also update pubspec.yaml to set the correct app version number and build number before each release.

Android — APK and App Bundle

There are two Android output formats. Use .apk for direct distribution (side-loading). Use .aab (App Bundle) for Google Play Store — it is smaller and lets Play Store generate optimised APKs for each device.

# Build a release APK (for direct installation)
flutter build apk --release

# Output: build/app/outputs/flutter-apk/app-release.apk
# Share this .apk file to install directly on Android phones

# Build split APKs — one per CPU architecture (smaller file per device)
flutter build apk --release --split-per-abi
# Output: app-arm64-v8a-release.apk (for most modern phones)
#         app-armeabi-v7a-release.apk (for older phones)
#         app-x86_64-release.apk (for emulators)

# Build App Bundle for Play Store submission
flutter build appbundle --release
# Output: build/app/outputs/bundle/release/app-release.aab
# Upload this .aab file to Google Play Console

iOS — IPA

⚠️ iOS builds require a Mac with Xcode

You cannot build an iOS app on Windows. You need a Mac with Xcode installed and an Apple Developer account ($99/year). If you don't have a Mac, you can use a Mac build service or a cloud Mac service like MacStadium.

# On Mac — build IPA for App Store

# Output: build/ios/ipa/Khan Chacha.ipa

A note from Raushan before we start

From Raushan — To every student in this batch

We built this application using Claude as our co-programmer. Claude is an extremely capable tool — it accelerated our development by weeks. But I need you to understand something clearly: AI does not automatically make your code secure.

Claude generates code that works. But "it works" and "it is secure" are two completely different things. Claude builds what you ask for. If you ask it to build a login route, it will build one that logs users in. It will not automatically add bcrypt, JWT validation, rate limiting, input sanitisation, audit logging, and role enforcement unless you specifically ask for each one.

The purpose of this checklist is to make sure that before you hand this application to Khan Chacha — or any client — you have reviewed every security dimension yourself. Not delegated it to Claude. Not assumed it was handled. Reviewed it personally, line by line. Use Claude to fix what you find. But you must be the one who looks.

AI is your co-pilot, not your co-pilot who also does the pre-flight safety check. That part is yours.

Security checklist — review every item before going live

✗

Passwords stored as plain text

In our current system, the users table stores passwords as plain text strings. If the database is breached, every password is exposed immediately.

Fix: Install bcrypt (npm install bcryptjs). Hash passwords when users are created: bcrypt.hash(password, 10). Verify with bcrypt.compare(input, hash) instead of direct string comparison in routes/auth.js.
✗

No authentication tokens (JWT)

After login, the app calls API endpoints without any proof of identity. Anyone who knows the API URL can call GET /api/requests and read all Khan Chacha maintenance data without logging in first.

Fix: Install jsonwebtoken (npm install jsonwebtoken). After successful login, return a signed JWT token. Add a middleware function to every protected route that verifies the token. Without a valid token, return HTTP 401. In Flutter, store the token in memory and send it as a header: Authorization: Bearer [token].
!

Role enforcement only in Flutter (client-side)

Staff users should only see their outlet's requests. This rule is currently enforced in Flutter — the app decides what to fetch based on the user's role. A technical user could call the API directly and bypass this completely.

Fix: Add role enforcement in the API routes. When a request arrives with a JWT token, extract the user's role from the token payload. If role is 'staff', automatically add a WHERE outlet_id = $user_outlet_id to every query — regardless of what the client requested.
✓

SQL injection — parameterized queries

All our API routes use parameterized queries ($1, $2 placeholders) throughout. User input is never concatenated directly into SQL strings.

Status: Already handled correctly. Verify by searching all route files for any SQL string that includes req.body or req.params directly — there should be none. Every value from the user must go through a $N parameter.
✗

No HTTPS in production

The app currently calls http:// URLs. HTTP means all data travels unencrypted — credentials, request details, outlet information — visible to anyone who can intercept the network traffic.

Fix: When deploying the Node.js API to AWS, attach a domain and use AWS Certificate Manager to provision a free SSL certificate. Configure your load balancer or Nginx to terminate HTTPS and forward to the Node.js app. Change the Flutter BASE_URL to https://.
!

CORS open to all origins

The API currently uses app.use(cors()) which allows any website on the internet to call the API from a browser. This is acceptable during development but not in production.

Fix: In production server.js, restrict CORS to known origins: app.use(cors({ origin: ['https://admin.khanchacha.com'], methods: ['GET','POST','PUT','PATCH','DELETE'] })) Note: CORS only applies to browser clients. The Flutter mobile/desktop app is not a browser — it is not affected by CORS restrictions.
✗

Detailed error messages exposed to clients

Our routes return res.status(500).json({ error: error.message }). In production, this can expose database table names, column names, internal IP addresses, and stack traces to anyone who triggers an error.

Fix: In production, return a generic message: { error: 'Internal server error' }. Log the full error server-side using a logging library like Winston or Morgan. Set NODE_ENV=production and use it to switch between verbose (dev) and generic (prod) error responses.
!

No rate limiting

The API has no limit on how many requests can be made per second from one IP address. A simple script could make 10,000 login attempts per minute — a brute-force password attack.

Fix: Install express-rate-limit (npm install express-rate-limit). Apply a stricter limit to the login route specifically: const loginLimiter = rateLimit({ windowMs: 15 * 60 * 1000, max: 10 }) app.use('/api/auth/login', loginLimiter) 10 login attempts per 15 minutes per IP is reasonable for a staff app.
!

No input validation on the server

The API checks if required fields are present but does not validate their content. A request with category: "DROP TABLE users" would be accepted and stored (the parameterized query prevents SQL injection, but the invalid value still enters the database).

Fix: Add server-side validation using a library like Joi or express-validator. Validate that category is one of the allowed ENUM values, that request_date is a valid date, that string fields are within reasonable length limits. Return 400 for any request that fails validation.
✓

Secrets in .gitignore

The .env file containing database credentials is excluded from Git via .gitignore.

Status: Handled. Verify by running git status and confirming .env does not appear. Also check your Git history: git log --all --full-history -- .env — if it was ever committed, rotate the password immediately.
!

Dependencies with known vulnerabilities

npm packages can have published security vulnerabilities. Using an outdated package version means you may be running code with known, documented exploits.

Fix: Run npm audit in the API project. Fix critical and high severity issues with npm audit fix. Run flutter pub outdated in the Flutter project and update packages with known issues. Make dependency auditing part of your pre-release routine.
