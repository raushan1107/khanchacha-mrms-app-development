What exactly is Flutter?

Flutter is a free, open-source toolkit made by Google for building apps. You write your app once, in one language, and Flutter turns that into a real Android app, a real iOS app, a real Windows app, a real website — all from the same source code.

Analogy

Think of Flutter like a tailor who can stitch the exact same design of shirt in three different fabrics — cotton for summer, wool for winter, silk for a wedding — without you having to design the shirt three separate times. You design it once. Flutter decides how to "stitch" it for each platform (Android, iOS, Windows) using that platform's native materials underneath.

What makes Flutter unusual is how it does this. Most cross-platform tools (like older hybrid app frameworks) wrap a website inside an app shell, which always feels a bit sluggish and "not quite native." Flutter does something different: it draws every single pixel on the screen itself, using its own rendering engine called Skia (and on newer versions, Impeller). It doesn't ask Android or iOS to draw the buttons for it — it draws them itself, which is why a Flutter app on Android and the same app on iPhone look pixel-for-pixel identical, and why it feels just as smooth as a native app.

What is Dart, and why does Flutter use it?

Dart is the programming language you'll actually type. It was also built by Google, specifically designed to pair with Flutter. If you've never written Dart before, here's the important reassurance: Dart is not Python, even though some of the syntax looks friendly like Python. Dart is a strongly-typed, compiled language — closer in spirit to Java or TypeScript than to Python. You'll see semicolons, curly braces, and explicit types, just written in a clean, modern style.

Analogy

If Flutter is the toolkit (the hammer, the saw, the blueprint system), Dart is the language those blueprints are written in. You can't separate them in practice — learning Flutter always means learning a bit of Dart along the way, the same way learning ASP.NET always means knowing some C#.

Why is Flutter a good choice for a project like Khan Chacha?

For a real business — a restaurant ordering app, a retail/e-commerce storefront, a small SaaS tool — the question is almost never "which language is technically superior." It's "what gets us a working app on both Android and iPhone, fastest, with the smallest team, without it looking cheap." Flutter wins that question for most small-to-mid businesses for four concrete reasons:

One team, one codebase

You don't need a separate Android developer and a separate iOS developer. One Flutter developer ships both. For a small business, that alone can cut development cost by close to half.

It looks and feels premium

Because Flutter draws its own UI, animations, transitions, and custom designs (like a branded ordering flow for Khan Chacha) look polished on both platforms without the "this was clearly built cheap" feeling some hybrid apps have.

Fast iteration during development

Hot reload (which you'll see in the Installation tab) means a developer can change a button's color and see it update on the phone in under a second — without restarting the whole app. This dramatically shortens the build-test-fix loop.

Strong AI tooling support

Because Flutter's code structure is predictable and well-documented, tools like Claude are excellent at generating, explaining, and modifying Flutter UI — which is exactly what today's session is built around.

"One codebase, both apps" — what does that actually mean?

When you ran flutter create earlier and saw folders named android, ios, windows appear, that confused a lot of people the first time — including your trainer when he started. Here is the clean way to think about it:

Analogy

Imagine you wrote one recipe for a cake. To actually bake it, you still need an oven — and ovens come in different brands and plug types. The android/ folder is essentially a tiny "Android-shaped plug adapter" that lets your one recipe (your Dart code in lib/) run inside an Android phone's oven. The ios/ folder is the same adapter, shaped for Apple's oven instead. You never touch the adapters. You only ever touch the recipe.

In practice this means: you will write 100% of your code inside the lib/ folder. The other platform folders exist quietly in the background, and Flutter uses them automatically the moment you ask it to build a real Android or iOS app.

If an app needs extremely deep integration with platform-specific hardware (certain AR/VR features, very specialized camera processing) or must shave off every last millisecond of performance, native development can still win. But for the vast majority of business apps — ordering systems, dashboards, internal tools, e-commerce — Flutter is the pragmatic, fast choice.

The one idea that unlocks everything: "Everything is a widget"

In Flutter, literally every visible thing on screen — and many invisible structural things too — is called a widget. A button is a widget. A row of text is a widget. The spacing around a button is a widget. Even the whole screen is one big widget made up of smaller widgets nested inside each other.

Analogy

Think of widgets like Russian nesting dolls (Matryoshka). The outermost doll is your whole screen. Inside it sits a slightly smaller doll (say, a list). Inside that sits an even smaller doll (one card in the list). Inside that, an even smaller doll (the text inside the card). When Claude generates Flutter code for you, what it's really doing is describing this entire nested-doll structure in one block of code — that's why the code often looks deeply indented, with brackets inside brackets inside brackets. It's not messy code; it's the nesting being made visible.

There are two flavors of widgets you'll meet constantly:

StatelessWidget

Shows information and never changes on its own. A name tag is a good comparison — once written, it doesn't update itself. Use this for things like a product card that just displays data it was given.

StatefulWidget

Has memory and can redraw itself when something changes. A live scoreboard is a good comparison — the number on it changes as the game progresses. Use this for filters, counters, forms — anything the user interacts with that needs to visually update.

What is the Flutter CLI, and what can it actually do?

CLI stands for Command Line Interface — it's simply the flutter word you type into a terminal followed by an instruction. You already met flutter create and flutter run. Here is the fuller toolbox, because there's quite a lot more it can do beyond just creating a project:

flutter create app_name
Scaffolds a brand-new project folder with everything needed to get started.
flutter run
Builds your app and launches it on whichever device or emulator is currently connected.
flutter pub get
Downloads any packages (third-party code libraries) listed in your project's dependency file.
flutter pub add package_name
Adds a new package to your project automatically — for example, a package for talking to a database.
flutter doctor
A full health check of your entire setup — tells you exactly what's missing or misconfigured.
flutter devices
Lists every device or emulator Flutter can currently see and could run your app on.
flutter emulators
Lists the virtual Android phones you've created, so you can start one without opening Android Studio.
flutter clean
Deletes old build files. This single command quietly fixes a large share of confusing build errors.
flutter build apk
Produces a real, installable Android app file (.apk) — the final product you'd share or publish.
flutter analyze
Scans your code for mistakes and bad practices before you even run it — like a spellchecker for code.
flutter test
Runs any automated tests you've written, to confirm your app behaves the way you expect.

What is an emulator, in plain terms?

An emulator is a fully virtual phone that runs as a window on your laptop. It isn't a picture or a video of a phone — it's an actual simulated Android device, with its own simulated screen, storage, and operating system, running entirely in software on your PC.

Analogy

It's similar to how a flight simulator lets a pilot practice flying without a real plane — except here, it's a phone simulator, and it behaves close enough to a real device that testing your app on it is a trustworthy substitute for testing on an actual phone, most of the time.

You create and manage these emulators using Android Studio's built-in AVD Manager (Android Virtual Device Manager). Once created, you don't need to keep opening Android Studio — you can launch that same emulator directly from a terminal or from VS Code afterward.

"But I'm not building for Android specifically — why do I need this?"

Because on a Windows laptop, there is no way to preview an iPhone app live (Apple only allows their iOS Simulator to run on a Mac). So even when your final target is iOS, or both platforms, the Android emulator becomes your everyday "see it live" tool during development on Windows. The Dart code you write is the same regardless of which emulator you're previewing it on — the visual result is what changes per platform, not your source code.

The folder structure, explained without jargon

When you ran flutter create, here's everything that appeared, and which of it you'll actually need to think about:

my_app/
├── lib/              ← YOUR CODE LIVES HERE. Almost everything you do happens in this folder.
│   └── main.dart       ← the starting point of your app
├── android/            ← auto-generated. You will basically never open this.
├── ios/                ← auto-generated. You will basically never open this.
├── windows/, web/, etc. ← same idea — generated wrappers for each platform.
├── test/               ← where automated tests live, if you write any.
├── pubspec.yaml      ← lists which third-party packages your app uses.
└── pubspec.lock        ← records the exact package versions actually installed.

The one rule that removes most of the confusion

If a folder's name matches an operating system (android, ios, windows, macos, linux, web) — it's generated scaffolding, and you can ignore it completely. Everything you write by hand goes inside lib/. That single rule resolves almost all the "which folder do I touch" confusion.

What's inside pubspec.yaml?

This file is your project's master list of dependencies and basic metadata — its name, version, and which external packages it relies on. You'll edit this file by hand occasionally, especially when adding a new capability (like talking to a database).

name: my_app
description: A maintenance tracking app

dependencies:
  flutter:
    sdk: flutter
  http: ^1.2.0        # lets the app talk to web APIs
  intl: ^0.19.0       # lets the app format dates nicely

After you add a new line here, you run flutter pub get to actually download it — the same way you'd restore packages in any other ecosystem.

What actually happens the moment you click Run

1
Your Dart code gets compiled
Everything inside lib/ is translated from human-readable Dart into a form the machine can run quickly.
2
Flutter's rendering engine gets attached to the platform shell
The "Android adapter" folder from earlier gets combined with Flutter's drawing engine, so your app knows how to actually paint pixels on that specific device.
3
A real installable app file gets produced
Everything bundles into one file — on Android this is an .apk — ready to be installed on a device.
4
That file gets pushed onto the emulator and launched
Just like installing any app — except it's happening automatically, onto your virtual phone, in the background.
5
Your first screen appears, and hot reload takes over
From this point on, every time you save a code change, Flutter quietly injects just that change into the running app — without redoing steps 1 through 4. That's why changes appear in under a second.

VS Code vs Android Studio — when to use which

Visual Studio Code

Lightweight, fast, and what you'll use for actually writing your Dart code day to day. With the Flutter extension installed, it gives you code completion, inline errors, a Run/Debug button, and a device picker. This is where 90% of your time will be spent.

Use it for: writing code, running the app, hot reload, debugging.

Android Studio

Heavier, originally built for native Android development. You mainly go here for two specific jobs: the AVD Manager (to create and manage emulators) and the SDK Manager (to install Android SDK components, including the command-line tools you needed during installation).

Use it for: one-time setup, creating emulators, managing SDK versions. Not for daily coding.

Important: there's no "project file" lock-in

Unlike some development environments where opening a project in a different tool means a different project format, a Flutter project is just a normal folder. Opening it in VS Code or in Android Studio is purely a choice of which window you want to look at it through — the underlying project doesn't change or get converted.

Installation guide — Windows, macOS, and Linux

Follow the steps for your operating system in order. Don't skip ahead even if a step looks optional — each step depends on the previous one being correctly finished.

Download and extract the Flutter SDK
Go to docs.flutter.dev/get-started/install/windows and download the latest stable SDK as a zip file. Extract it to a simple path with no spaces and no special characters:

Reading Dart and Flutter code — what you're actually looking at

You don't need to memorize how to write Dart from scratch. Your job during this training is to read and confidently modify what Claude generates for you. This page covers the patterns you'll see constantly, written plainly enough that the syntax stops feeling unfamiliar.

First, an important clarification

Dart often gets compared to Python because both can look clean and readable. But Dart is not Python. Dart requires semicolons at the end of most lines, uses curly braces { } to group code, and is strongly typed — meaning you usually declare what kind of value a variable holds. If you've seen any Java, C#, JavaScript, or TypeScript before, Dart will feel much closer to those.

The entry point: every Flutter app starts the same way

Open any main.dart file and you'll always find this shape at the top:

void main() {
  runApp(const MyApp());
}

Read this exactly as it sounds: "When the app starts, build and display MyApp." The word void simply means this function doesn't hand back a result to anything — it just performs an action. Everything your app does, ultimately, traces back to this one line.

Classes — the building blocks of every screen

A widget is written as a class. If "class" feels like a heavy word, think of it simply as a labeled container that bundles together some data and the instructions for what to do with that data.

class RequestCard extends StatelessWidget {
  final String productName;
  final String status;

  const RequestCard({super.key, required this.productName, required this.status});

  @override
  Widget build(BuildContext context) {
    return Card(child: Text(productName));
  }
}

Breaking this down piece by piece:

  • class RequestCard extends StatelessWidget — "I'm creating something called RequestCard, and it behaves like a Stateless widget (it just displays things, doesn't change on its own)."
  • final String productName; — "This card will always have a piece of text called productName, and once it's set, it can't be changed." The word final means "locked after being set once."
  • const RequestCard({...}) — this is the constructor: the recipe for how to create one of these cards, listing what information it needs to be handed.
  • required this.productName — "You must give me a productName when you create this card — it's not optional."
  • Widget build(BuildContext context) — every widget has this method. It's the part that actually describes what should appear on screen.

Variables — three ways you'll see them declared

var name = 'Air Conditioner';       // Dart figures out the type automatically
String name = 'Air Conditioner';    // you stated the type explicitly: a String (text)
final id = 'REQ-001';             // set once, can never be changed afterward

You'll see String (text), int (whole numbers), double (decimal numbers), and bool (true or false) most often as types.

Lists and maps — Dart's version of arrays and dictionaries

// A List — an ordered collection of items, same idea as an array
List<String> statuses = ['Open', 'Pending', 'Closed'];

// A Map — pairs of keys and values, same idea as a dictionary
Map<String, Color> statusColors = {
  'Open': Colors.green,
  'Closed': Colors.red,
};

String interpolation — inserting variables into text

Instead of manually joining pieces of text together, Dart lets you drop a variable directly inside a string using a dollar sign:

final name = 'AC Filter';
final count = 3;

Text('Request: $name')                  // → "Request: AC Filter"
Text('Total: ${count + 1} items')       // → "Total: 4 items" — curly braces let you run a calculation inline

Functions — including the short "arrow" style

// A normal function, written out fully
Color getColor(String status) {
  if (status == 'Open') return Colors.green;
  return Colors.red;
}

// The same function, written using arrow shorthand — common in Flutter code
Color getColor(String s) => s == 'Open' ? Colors.green : Colors.red;

That => arrow simply means "this function's entire body is just this one expression — skip the curly braces and the word return." You'll see this constantly in Claude-generated code, especially for short, simple functions.

Named parameters — the curly braces inside function calls

This is one of the patterns that confuses newcomers most, because it looks unusual at first:

class StatusChip extends StatelessWidget {
  final String label;     // required — must always be provided
  final Color? color;    // optional — the ? means "this can be left empty"

  const StatusChip({
    super.key,
    required this.label,
    this.color,
  });
}

The curly braces { } inside the constructor mean "these parameters are named" — meaning when you actually use this widget elsewhere, you must write the parameter's name before its value, like StatusChip(label: 'Open', color: Colors.green) rather than just listing values in order. This is Flutter's way of making widget code self-explanatory just by reading it, even with dozens of parameters.

Putting it all together — a complete, small, real widget

Here is a complete, working widget combining everything above. Read it top to bottom now that you have the vocabulary:

class RequestCard extends StatelessWidget {
  final String productName;
  final String date;
  final String status;

  const RequestCard({
    super.key,
    required this.productName,
    required this.date,
    required this.status,
  });

  Color get statusColor {
    if (status == 'Open') return Colors.green;
    if (status == 'Pending') return Colors.amber;
    return Colors.red;
  }

  @override
  Widget build(BuildContext context) {
    return Card(
      child: ListTile(
        title: Text(productName),
        subtitle: Text(date),
        trailing: Chip(
          label: Text(status),
          backgroundColor: statusColor,
        ),
      ),
    );
  }
}

If you can now read this and explain in plain English what each line does, you're ready to start modifying Claude's output confidently — which is exactly what the next tab walks you through.

A note before you start — claude.ai vs Claude Code, and why this tab uses Claude Code

You can absolutely use claude.ai (the chat) for every prompt below — copy the prompt, paste it into chat, copy the code block back out, paste it into the right file in VS Code yourself. That manual loop works and is genuinely useful while you're still learning to read Dart.

The split that works best in practice: use Claude Code for the actual building — the prompts below. Building and learning are different modes — Claude Code is your hands, claude.ai is your tutor.

How approving changes works

After you send a prompt, Claude Code will explain what it's about to do, then show you the exact file content it wants to write — like a before/after diff. It'll ask something like "Do you want to make this edit?" with options to approve. Read what it's about to change, then approve. This is the safety behavior the warning below is about — keep it on.

About "dangerously skip permissions"

There's a flag, --dangerously-skip-permissions, that turns off these confirmation prompts entirely, letting Claude Code edit files without asking each time.

For this training: do not use this flag. Keep every confirmation on. Working with real data — connecting Flutter to a database

You've already studied databases before this training, so this tab focuses specifically on the one piece you haven't done yet: how a Flutter app actually fetches data from a database and shows it on screen, and how Claude can help you build that connection.

The fundamental shape of every data-driven app screen

No matter which database you use — SQL Server, PostgreSQL, Firebase, or anything else — the pattern your Flutter app follows is always the same three steps:

1
Ask for the data
Your Flutter app sends a request — usually over the internet, to an API — asking "give me the list of maintenance requests."
2
Wait, because this takes time
Network requests are never instant. While waiting, your screen shows a loading spinner instead of nothing — this is exactly what the FutureBuilder pattern from the Hands-On tab was rehearsing.
3
Convert the response into widgets
The data usually arrives as JSON (a structured text format). Your app converts that JSON into Dart objects (like the MenuItem model class), then builds a card widget for each one.

Analogy

Think of your Flutter app as a waiter at a restaurant, and the database as the kitchen. The waiter (your app) doesn't cook the food (store the data) — they walk to the kitchen, ask for what the customer ordered, wait while it's prepared, then bring it back and present it nicely on a plate (the widgets on screen). The waiter is never directly reaching into the fridge — there's always a clear request-and-response step in between.

Where does the "asking" actually happen?

This middle layer — the thing your app actually talks to — is usually called an API (Application Programming Interface). Your database almost never talks directly to your phone app for security and structural reasons. Instead:

Flutter App  →  API (a web server)  →  Database
   (phone)        (the "waiter's          (the "kitchen,"
                   kitchen window")        where data lives)

In Flutter, the most common package for talking to an API is simply called http — the same package you saw mentioned in the pubspec.yaml example earlier in this guide.

A real, minimal example: fetching a list from an API

Here's what that "ask, wait, convert" pattern actually looks like in code, kept deliberately simple:

import 'package:http/http.dart' as http;
import 'dart:convert';

Future<List<MenuItem>> fetchMenuItems() async {
  final response = await http.get(Uri.parse('https://your-api.com/menu'));

  if (response.statusCode == 200) {
    List data = jsonDecode(response.body);
    return data.map((item) => MenuItem.fromJson(item)).toList();
  } else {
    throw Exception('Failed to load menu');
  }
}

Notice the words async and await — these mean "this function involves waiting for something, so pause here until the answer comes back, without freezing the rest of the app." This is conceptually identical to async/await in most modern programming languages, just written with Dart's syntax.

Where Claude genuinely helps the most with database work

Writing the model class

Given a sample of your real JSON data, Claude can generate the exact Dart model class (like MenuItem) with a correct fromJson method, saving you from manually mapping every field by hand.

Writing the fetch logic

Claude can write the full async function that calls your specific API endpoint, handles errors gracefully, and converts the response — exactly the pattern shown above, tailored to your actual data shape.

Connecting it to your UI

Claude can wire your existing HomeScreen to use FutureBuilder with this real fetch function instead of simulated hardcoded data — swapping fake data for real data with minimal changes to the screen itself.

Explaining unfamiliar database services

If your project ends up using something like Firebase or Supabase instead of a traditional SQL database, Claude can explain how that specific service's Flutter package works, since the basic "ask, wait, convert" shape stays the same either way.

Questions and discussion

A running collection of the exact questions raised during this training, answered the same way they were answered live — so you can revisit them anytime, or share this page with someone who missed the session.

What is Dart, really, and is it like Python?▾
Dart is the programming language behind Flutter, created by Google specifically to pair with it. Despite looking clean and approachable, Dart is not Python — it requires semicolons, uses curly braces to group code, and is strongly typed. If you've seen Java, C#, JavaScript, or TypeScript, Dart will feel far more familiar than Python does.
Why does flutter create generate android, ios, and windows folders even if I only care about one platform?▾
Because Flutter's whole value proposition is writing once and running everywhere. Each of those folders is a thin native "adapter" that lets your one set of Dart code run inside that specific platform's environment. You never edit these folders directly — all of your actual work happens inside the lib folder, regardless of which platform you eventually ship to.
If I'm not targeting Android, why do I need Android Studio at all?▾
Android Studio isn't there because you're building an Android app — it's there because it's the easiest way to get the Android SDK and a virtual phone (emulator) onto your computer. On Windows or Linux, there's no other live-preview option besides the Android emulator (Apple only allows their iOS Simulator to run on a Mac). So even iOS-focused developers on Windows install Android Studio purely for its emulator and SDK tools, then write 100% of their actual code in VS Code.
What exactly is an emulator?▾
An emulator is a fully virtual phone running as a window on your computer — not a picture or video, but an actual simulated Android device with its own screen, storage, and operating system, running entirely in software. You create and manage these through Android Studio's Device Manager, and once created, you can launch and use them directly from VS Code or the terminal without reopening Android Studio.
What is the difference between VS Code and Android Studio for daily work?▾
VS Code is lightweight and is where you'll write your Dart code, run your app, and use hot reload — this is where 90% of your time will be spent. Android Studio is heavier and mainly useful for two setup tasks: creating/managing emulators (Device Manager) and installing Android SDK components (SDK Manager). There's no project format lock-in either way — both tools simply open the same project folder.
What does "hot reload" actually mean, and how is it different from "hot restart"?▾
Hot reload takes your latest code change and injects just that change into the already-running app, keeping all of your current screen state intact — this is why it's near-instant. Hot restart fully restarts the app from scratch, clearing all state, which is occasionally necessary if a change doesn't seem to be taking effect through hot reload alone (for example, changes to certain low-level setup code).
Why does main.dart call two other methods/classes instead of doing everything itself?▾
This is intentional separation of responsibilities, not unnecessary complexity. main() simply starts the engine. The next class (commonly MyApp) configures the whole app — its theme, title, and which screen to show first. The class after that (commonly MyHomePage or HomeScreen) is the actual screen content the user interacts with. Splitting these apart means each piece can be understood, tested, and changed independently.
When should I use Claude (the chat) versus Claude Code (the terminal tool)?▾
Use Claude chat when you're learning a concept, want to ask "why" something works a certain way, need help fixing an error message, or want feedback on a UI decision — anywhere a conversation helps more than a direct file edit. Use Claude Code when you already know what feature you want built and you'd rather it land directly in your project files without you copy-pasting. Most experienced Flutter+Claude workflows end up using both in the same session: Claude Code to build, Claude chat to understand and troubleshoot.
Should I ever use the "dangerously skip permissions" flag with Claude Code?▾
Not while you're learning. That flag turns off Claude Code's confirmation prompts before it edits or deletes files, which removes an important safety check and an important learning opportunity — seeing exactly what's about to change. It's intended for experienced developers working in fully disposable, sandboxed environments, not for students working on their personal laptop during a training session.
Why does my app need a loading spinner just to show a list of data?▾
Because fetching data from a database or API always takes some amount of time — even a fast connection has a noticeable delay compared to instantly displaying something already in memory. Showing a loading spinner during that gap (using a pattern called FutureBuilder) is simply honest, good UX — it tells the user "something is happening," rather than leaving the screen blank and looking broken.
My code has a red error in the terminal. What's the fastest way to fix it with Claude's help?▾
Copy the full error text exactly as it appears, and paste it to Claude along with the relevant file's current content. Claude can almost always pinpoint the exact cause — commonly a missing import statement, a typo in a file path, or a widget being given a parameter it doesn't expect — and will give you the corrected code directly. This error-paste-fix loop is the real, everyday workflow of working with AI assistance.
Do I need to memorize Dart syntax to be productive with Flutter and Claude?▾
No — and that's by design for this training. Your actual job is to read Flutter code confidently enough to understand what Claude generates, recognize when something looks wrong, and describe clearly what change you want next. Memorizing syntax becomes useful later as you get more hands-on practice, but it is not a prerequisite for being productive starting today.
