Getting Started Without Getting Lost

A hands-on Flutter guide built for students with zero Flutter background — using Claude as your AI pair-programmer from day one.

What exactly is Flutter?

Flutter is a free, open-source toolkit made by Google for building apps. You write your app once, in one language, and Flutter turns that into a real Android app, a real iOS app, a real Windows app, a real website — all from the same source code.

Analogy

Think of Flutter like a tailor who can stitch the exact same design of shirt in three different fabrics — cotton for summer, wool for winter, silk for a wedding — without you having to design the shirt three separate times. You design it once. Flutter decides how to "stitch" it for each platform (Android, iOS, Windows) using that platform's native materials underneath.

What makes Flutter unusual is how it does this. Most cross-platform tools (like older hybrid app frameworks) wrap a website inside an app shell, which always feels a bit sluggish and "not quite native." Flutter does something different: it draws every single pixel on the screen itself, using its own rendering engine called Skia (and on newer versions, Impeller). It doesn't ask Android or iOS to draw the buttons for it — it draws them itself, which is why a Flutter app on Android and the same app on iPhone look pixel-for-pixel identical, and why it feels just as smooth as a native app.

What is Dart, and why does Flutter use it?

Dart is the programming language you'll actually type. It was also built by Google, specifically designed to pair with Flutter. If you've never written Dart before, here's the important reassurance: Dart is not Python, even though some of the syntax looks friendly like Python. Dart is a strongly-typed, compiled language — closer in spirit to Java or TypeScript than to Python. You'll see semicolons, curly braces, and explicit types, just written in a clean, modern style.

Analogy

If Flutter is the toolkit (the hammer, the saw, the blueprint system), Dart is the language those blueprints are written in. You can't separate them in practice — learning Flutter always means learning a bit of Dart along the way.

Why is Flutter a good choice for an internal tool like this?

Khan Chacha doesn't need a public storefront app for this project — what's needed is an internal maintenance tracking tool: a small number of managers logging and tracking equipment or facility maintenance requests, on their own phones. For exactly this kind of internal business tool, Flutter is a strong, practical choice for four reasons:

One developer, one codebase

You don't need a separate Android developer and a separate iOS developer for a small internal tool. One person can build and maintain it for whichever phones the managers actually carry.

Looks professional with little effort

Even an internal tool benefits from feeling clean and trustworthy — color-coded statuses, clear cards, a simple form — and Flutter makes that easy without a design team.

Fast iteration during development

Hot reload means a developer can change a button's color and see it update on the phone in under a second — without restarting the whole app. This dramatically shortens the build-test-fix loop, which matters a lot when you're learning.

Strong AI tooling support

Because Flutter's code structure is predictable and well-documented, tools like Claude are excellent at generating, explaining, and modifying Flutter UI — which is exactly what this training is built around.

"One codebase, many apps" — what does that actually mean?

When you ran flutter create and saw folders named android, ios, windows appear, that confused a lot of people the first time. Here is the clean way to think about it:

Analogy

Imagine you wrote one recipe for a cake. To actually bake it, you still need an oven — and ovens come in different brands and plug types. The android/ folder is essentially a tiny "Android-shaped plug adapter" that lets your one recipe (your Dart code in lib/) run inside an Android phone's oven. The ios/ folder is the same adapter, shaped for Apple's oven instead. You never touch the adapters. You only ever touch the recipe.

In practice this means: you will write 100% of your code inside the lib/ folder. The other platform folders exist quietly in the background, and Flutter uses them automatically the moment you ask it to build a real Android or iOS app.

If an app needs extremely deep integration with platform-specific hardware, or must shave off every last millisecond of performance, native development can still win. But for an internal business tool like this maintenance tracker, Flutter is the pragmatic, fast choice.

The one idea that unlocks everything: "Everything is a widget"

In Flutter, literally every visible thing on screen — and many invisible structural things too — is called a widget. A button is a widget. A row of text is a widget. The spacing around a button is a widget. Even the whole screen is one big widget made up of smaller widgets nested inside each other.

Analogy

Think of widgets like Russian nesting dolls (Matryoshka). The outermost doll is your whole screen. Inside it sits a slightly smaller doll (say, a list). Inside that sits an even smaller doll (one card in the list). Inside that, an even smaller doll (the text inside the card). When Claude generates Flutter code for you, what it's really doing is describing this entire nested-doll structure in one block of code — that's why the code often looks deeply indented, with brackets inside brackets inside brackets. It's not messy code; it's the nesting being made visible.

There are two flavors of widgets you'll meet constantly:

StatelessWidget

Shows information and never changes on its own. A name tag is a good comparison — once written, it doesn't update itself. Use this for things like a request card that just displays data it was given.

StatefulWidget

Has memory and can redraw itself when something changes. A live scoreboard is a good comparison — the number on it changes as the game progresses. Use this for filters, forms — anything the user interacts with that needs to visually update.

What is the Flutter CLI, and what can it actually do?

CLI stands for Command Line Interface — it's simply the flutter word you type into a terminal followed by an instruction. Here is the fuller toolbox:

flutter create app_name
Scaffolds a brand-new project folder with everything needed to get started.
flutter run
Builds your app and launches it on whichever device or emulator is currently connected.
flutter pub get
Downloads any packages (third-party code libraries) listed in your project's dependency file.
flutter pub add package_name
Adds a new package to your project automatically — for example, a package for talking to a database.
flutter doctor
A full health check of your entire setup — tells you exactly what's missing or misconfigured.
flutter devices
Lists every device or emulator Flutter can currently see and could run your app on.
flutter emulators
Lists the virtual Android phones you've created, so you can start one without opening Android Studio.
flutter clean
Deletes old build files. This single command quietly fixes a large share of confusing build errors.
flutter build apk
Produces a real, installable Android app file (.apk) — the final product you'd share or publish.
flutter analyze
Scans your code for mistakes and bad practices before you even run it.
flutter test
Runs any automated tests you've written, to confirm your app behaves the way you expect.

What is an emulator, in plain terms?

An emulator is a fully virtual phone that runs as a window on your laptop — an actual simulated Android device, with its own simulated screen, storage, and operating system, running entirely in software on your PC.

Analogy

It's similar to how a flight simulator lets a pilot practice flying without a real plane — except here, it's a phone simulator, and it behaves close enough to a real device that testing your app on it is a trustworthy substitute for testing on an actual phone, most of the time.

You create and manage these emulators using Android Studio's built-in Device Manager. Once created, you can launch that same emulator directly from a terminal or from VS Code afterward.

"But I'm not building for Android specifically — why do I need this?"

On a Windows laptop, there is no way to preview an iPhone app live (Apple only allows their iOS Simulator to run on a Mac). So the Android emulator becomes your everyday "see it live" tool during development on Windows, regardless of which platform you eventually ship to.

The folder structure, explained without jargon

When you ran flutter create, here's everything that appeared, and which of it you'll actually need to think about:

my_app/
├── lib/              ← YOUR CODE LIVES HERE. Almost everything you do happens in this folder.
│   └── main.dart       ← the starting point of your app
├── android/            ← auto-generated. You will basically never open this.
├── ios/                ← auto-generated. You will basically never open this.
├── windows/, web/, etc. ← same idea — generated wrappers for each platform.
├── test/               ← where automated tests live, if you write any.
├── pubspec.yaml      ← lists which third-party packages your app uses.
└── pubspec.lock        ← records the exact package versions actually installed.

The one rule that removes most of the confusion

If a folder's name matches an operating system (android, ios, windows, macos, linux, web) — it's generated scaffolding, and you can ignore it completely. Everything you write by hand goes inside lib/.

What's inside pubspec.yaml?

This file is your project's master list of dependencies and basic metadata. You'll edit this file by hand occasionally, especially when adding a new capability.

name: my_app
description: A maintenance tracking app

dependencies:
  flutter:
    sdk: flutter
  http: ^1.2.0        # lets the app talk to web APIs
  intl: ^0.19.0       # lets the app format dates nicely

After you add a new line here, you run flutter pub get to actually download it.

What actually happens the moment you click Run

1
Your Dart code gets compiled
Everything inside lib/ is translated from human-readable Dart into a form the machine can run quickly.
2
Flutter's rendering engine gets attached to the platform shell
The "Android adapter" folder from earlier gets combined with Flutter's drawing engine, so your app knows how to actually paint pixels on that specific device.
3
A real installable app file gets produced
Everything bundles into one file — on Android this is an .apk — ready to be installed on a device.
4
That file gets pushed onto the emulator and launched
Just like installing any app — except it's happening automatically, onto your virtual phone, in the background.
5
Your first screen appears, and hot reload takes over
From this point on, every time you save a code change, Flutter quietly injects just that change into the running app — without redoing steps 1 through 4. That's why changes appear in under a second.

VS Code vs Android Studio — when to use which

Visual Studio Code

Lightweight, fast, and what you'll use for actually writing your Dart code day to day. With the Flutter extension installed, it gives you code completion, inline errors, a Run/Debug button, and a device picker. This is where 90% of your time will be spent.

Use it for: writing code, running the app, hot reload, debugging.

Android Studio

Heavier, originally built for native Android development. You mainly go here for two specific jobs: the Device Manager (to create and manage emulators) and the SDK Manager (to install Android SDK components).

Use it for: one-time setup, creating emulators, managing SDK versions. Not for daily coding.

Reading Dart and Flutter code — what you're actually looking at

You don't need to memorize how to write Dart from scratch. Your job during this training is to read and confidently modify what Claude generates for you.

First, an important clarification

Dart often gets compared to Python because both can look clean. But Dart is not Python. Dart requires semicolons, uses curly braces { } to group code, and is strongly typed. If you've seen Java, C#, JavaScript, or TypeScript, Dart will feel much closer to those.

The entry point: every Flutter app starts the same way

void main() {
  runApp(const MyApp());
}

Read this exactly as it sounds: "When the app starts, build and display MyApp." Everything your app does, ultimately, traces back to this one line.

Classes — the building blocks of every screen

class RequestCard extends StatelessWidget {
  final String productName;
  final String status;

  const RequestCard({super.key, required this.productName, required this.status});

  @override
  Widget build(BuildContext context) {
    return Card(child: Text(productName));
  }
}
  • class RequestCard extends StatelessWidget — "I'm creating something called RequestCard, and it just displays things, doesn't change on its own."
  • final String productName; — "This card will always have a piece of text called productName, locked once set."
  • const RequestCard({...}) — the constructor: the recipe for how to create one of these cards.
  • required this.productName — "You must give me a productName — it's not optional."
  • Widget build(BuildContext context) — every widget has this method. It describes what should appear on screen.

Variables — three ways you'll see them declared

var name = 'AC Filter';       // Dart figures out the type automatically
String name = 'AC Filter';    // you stated the type explicitly
final id = 'REQ-001';        // set once, can never be changed afterward

Lists and maps — Dart's version of arrays and dictionaries

// A List — an ordered collection, same idea as an array
List<String> statuses = ['Open', 'Pending', 'Closed'];

// A Map — key/value pairs, same idea as a dictionary
Map<String, Color> statusColors = {
  'Open': Colors.green,
  'Closed': Colors.red,
};

String interpolation — inserting variables into text

final name = 'AC Filter';
Text('Request: $name')   // → "Request: AC Filter"

Named parameters — the curly braces inside function calls

class StatusChip extends StatelessWidget {
  final String label;
  final Color? color;    // ? means optional

  const StatusChip({
    super.key,
    required this.label,
    this.color,
  });
}

The curly braces { } inside the constructor mean "these parameters are named" — you write the parameter's name before its value, like StatusChip(label: 'Open', color: Colors.green), rather than just listing values in order.

If you can now read this and explain in plain English what each line does, you're ready for the next two tabs — your first guided app, then building the maintenance tracker piece by piece with Claude Code.

Step 2 — open the code and find what controls what you saw

In VS Code, open lib/main.dart. This single file is responsible for everything you just saw on screen. Look for these specific pieces:

The title text

Search for the word title: — appears twice, once for the app's internal title, once for the text shown in the AppBar at the top of the screen.

The theme color

Search for seedColor: — this single value controls the overall color scheme of the entire app.

The counter number

Search for _counter — this is the variable holding the number you saw increase with each tap.

The + button

Search for FloatingActionButton — this is the round button in the corner, and onPressed: right above it is what runs every time you tap it.

Building the maintenance tracker with Claude Code

Now that you've seen what Flutter generates by default and made a few manual edits yourself, this is where Claude Code starts doing the heavier structural work — creating new files, organizing folders, wiring screens together — while you stay in control of every change.

A note before you start — claude.ai vs Claude Code, and why this tab uses Claude Code

You can use claude.ai (the chat) for every prompt below too — copy the prompt, paste it into chat, copy the code block back out, paste it into the right file in VS Code yourself. That manual loop works and is genuinely useful while you're still learning to read Dart.

This tab uses Claude Code instead because building the maintenance tracker means creating and editing multiple files across multiple folders — models, screens, widgets, constants. Doing that by hand means many copy-paste round trips, and one mistyped folder name breaks the whole thing. Claude Code sees your actual project, creates the right files in the right folders itself, and shows you the exact change as a diff before applying it. The split that works well: Claude Code for the actual building (the prompts below), claude.ai open in a separate tab for understanding why something works, fixing red terminal errors, getting UI feedback, or refining a prompt before you run it.

How approving changes works

After a prompt, Claude Code explains what it's about to do, then shows the exact file content as a diff and asks you to approve. Read it, then approve. Keep this on — it's the safety check you want while learning.

About "dangerously skip permissions"

There's a flag, --dangerously-skip-permissions, that turns off these confirmations entirely. Do not use it for this training. Keep every confirmation on while you're still learning what Flutter code looks like — seeing each change is part of the learning, not friction to remove.

Working with real data — connecting Flutter to a database

This tab focuses on one piece you haven't done yet: how a Flutter app actually fetches data from a database and shows it on screen, and how Claude can help build that connection — conceptually, without wiring up a live backend during this session.

The fundamental shape of every data-driven app screen

No matter which database is used, the pattern a Flutter app follows is always the same three steps:

1
Ask for the data
The app sends a request — usually over the internet, to an API — asking "give me the list of maintenance requests."
2
Wait, because this takes time
Network requests are never instant. While waiting, the screen shows a loading spinner instead of nothing.
3
Convert the response into widgets
The data arrives as JSON. The app converts that into Dart objects (like the MaintenanceRequest model), then builds a card widget for each one.

Analogy

Think of the app as a waiter, and the database as the kitchen. The waiter doesn't store the food — they walk to the kitchen, ask for the order, wait while it's prepared, then bring it back and present it on a plate (the widgets on screen). There's always a clear request-and-response step in between.

A real, minimal example: fetching a list from an API

import 'package:http/http.dart' as http;
import 'dart:convert';

Future<List<MaintenanceRequest>> fetchRequests() async {
  final response = await http.get(Uri.parse('https://your-api.com/requests'));

  if (response.statusCode == 200) {
    List data = jsonDecode(response.body);
    return data.map((item) => MaintenanceRequest.fromJson(item)).toList();
  } else {
    throw Exception('Failed to load requests');
  }
}

async and await mean "this involves waiting, so pause here until the answer comes back, without freezing the rest of the app."

Where Claude genuinely helps the most with database work

Writing the model class

Given a sample of real JSON data, Claude can generate the exact model class with a correct fromJson method.

Writing the fetch logic

Claude can write the full async function tailored to your actual API and data shape, including error handling.

Connecting it to your UI

Claude can wire your existing screen to use a FutureBuilder with real data instead of hardcoded sample data.

Explaining unfamiliar database services

If the real backend ends up being Firebase, Supabase, or a company SQL server, Claude can explain how that specific connection works.

Questions and discussion

A running collection of the exact questions raised during this training, answered the same way they were answered live.

What is Dart, really, and is it like Python?▾
Dart is the programming language behind Flutter, created by Google specifically to pair with it. Despite looking clean and approachable, Dart is not Python — it requires semicolons, uses curly braces to group code, and is strongly typed. If you've seen Java, C#, JavaScript, or TypeScript, Dart will feel far more familiar than Python does.
Why does flutter create generate android, ios, and windows folders even if I only care about one platform?▾
Because Flutter's whole value proposition is writing once and running everywhere. Each of those folders is a thin native "adapter" that lets your one set of Dart code run inside that specific platform's environment. You never edit these folders directly — all of your actual work happens inside the lib folder.
If I'm not targeting Android, why do I need Android Studio at all?▾
Android Studio isn't there because you're building an Android app — it's there because it's the easiest way to get the Android SDK and a virtual phone (emulator) onto your computer. On Windows, there's no other live-preview option besides the Android emulator. So even iOS-focused developers on Windows install Android Studio purely for its emulator and SDK tools, then write 100% of their actual code in VS Code.
What exactly is an emulator?▾
An emulator is a fully virtual phone running as a window on your computer — an actual simulated Android device with its own screen, storage, and operating system, running entirely in software. You create and manage these through Android Studio's Device Manager, and once created, you can launch and use them directly from VS Code or the terminal.
My emulator was working before but now flutter emulators shows nothing — what happened?▾
An AVD (virtual device) can disappear if it was deleted during troubleshooting, or if its system image got corrupted and was removed. The fix is simply to recreate it from scratch through Android Studio's Device Manager — Create Device, pick a phone, pick a system image, and this time watch the download progress bar reach 100% before clicking Finish. That download-completion step is the most common point of silent failure.
flutter --version says "not recognized" even though I added it to PATH — why?▾
Two common causes. First: the terminal you're testing in was opened before the PATH change was saved — PowerShell only reads PATH once at startup, so close that terminal completely and open a brand new one. Second: the PATH entry might not have actually saved — when editing Environment Variables, all three windows (Edit, Environment Variables, System Properties) need OK clicked, not Cancel, on every one of them. You can check what's currently active with `$env:Path -split ';' | Select-String "flutter"` — if that prints nothing, the PATH genuinely isn't set in that session.
What does "hot reload" actually mean, and how is it different from "hot restart"?▾
Hot reload takes your latest code change and injects just that change into the already-running app, keeping all current screen state intact — this is why it's near-instant. Hot restart fully restarts the app from scratch, clearing all state, occasionally necessary if a change doesn't take effect through hot reload alone.
When should I use Claude (the chat) versus Claude Code (the terminal tool)?▾
Use Claude chat when learning a concept, asking "why" something works a certain way, fixing an error message, getting UI feedback, or refining a prompt before running it. Use Claude Code when you already know what feature you want and would rather it land directly in your project files without manual copy-pasting. Most workflows use both in the same session: Claude Code to build, Claude chat to understand and troubleshoot.
Should I ever use the "dangerously skip permissions" flag with Claude Code?▾
Not while learning. That flag turns off Claude Code's confirmation prompts before it edits or deletes files, removing an important safety check and learning opportunity. It's intended for experienced developers in disposable, sandboxed environments, not for students on their own laptop during training.
My code has a red error in the terminal. What's the fastest way to fix it?▾
Copy the full error text exactly as it appears, and paste it to Claude along with the relevant file's content. Claude can almost always pinpoint the cause — commonly a missing import, a typo in a file path, a mismatched widget parameter, or (as happened during this training) an SDK-version type rename like CardTheme becoming CardThemeData — and gives the corrected code directly.
I got a Gradle "timeout waiting to lock build logic queue" error — what does that mean?▾
This means a previous build process is still running in the background and holding a lock file, usually because an earlier flutter run wasn't stopped cleanly. The error message tells you the exact process ID to end (taskkill /PID <number> /F on Windows), followed by flutter clean to clear any half-finished build state, then flutter run again.
Do I need to memorize Dart syntax to be productive with Flutter and Claude?▾
No — by design. The actual job is to read Flutter code confidently enough to understand what Claude generates, recognize when something looks wrong, and describe clearly what change you want next. Memorizing syntax becomes useful with more practice, but it isn't a prerequisite for being productive starting today.
