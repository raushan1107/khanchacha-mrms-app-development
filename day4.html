What is PostgreSQL — and why we chose it

PostgreSQL is a free, open-source relational database. "Relational" means data lives in tables that reference each other — just like a Khan Chacha office filing system where each maintenance request slip references the outlet file and the staff member file rather than copying all their details onto every slip.

We chose PostgreSQL over SQLite (single device only), MongoDB (wrong shape for structured data), Firebase (costly at scale, no SQL), and MySQL (PostgreSQL has better features). The same schema we built locally runs identically on Supabase, AWS RDS, or Azure — no rewriting.

The database architecture that was formed

After Day 3, three tables exist inside the PostgreSQL engine with this structure:

outlets (id, name, address, city, pincode)
   ↑ outlet_id FK
users (id, name, username, password, role, outlet_id)
   ↑ raised_by FK    ↑ outlet_id FK
maintenance_requests (id, product_name, category ENUM, status ENUM, outlet_id, raised_by, status_history JSONB, ...)
  • ENUM types enforce that category can only be Electrical, AC, or Kitchen — not "electircal" or anything else. The database rejects invalid values before they ever reach the app.
  • Foreign keys mean a maintenance request cannot reference an outlet that doesn't exist. Referential integrity is enforced at the database level — not in Flutter code.
  • Indexes on status, outlet_id, and request_date make the most common queries (filter by status, filter by outlet, sort by date) fast even with thousands of rows.
  • JSONB for status_history stores the change log as a JSON array in a single column — no separate table needed for this use case.

Why API layer comes immediately after database

The Bridge

Right now, the Flutter app talks to PostgreSQL directly using the postgres Dart package. The database password is a constant inside your Dart code. When you build the app into an APK and distribute it, that password travels with the APK. Anyone who decompiles the app gets your database credentials. For a training session on a local network, this is acceptable. For a real restaurant chain's production system — it is not.

The API layer is the bridge that fixes this. Flutter talks to the API (a Node.js server) over HTTPS. The API talks to PostgreSQL. The database password never leaves the server. Flutter never knows it. That's the problem Day 4 solves.

What is an API Layer?

API stands for Application Programming Interface. The "layer" part means it sits between your frontend app and your database — receiving requests from Flutter, talking to PostgreSQL, and sending back results. It is a separate program running on a server.

In one sentence

An API layer is a program that listens for requests from apps, applies business logic and security rules, asks the database for data, and sends the answer back — so apps never touch the database directly.

The restaurant analogy — Khan Chacha itself

Real World Analogy

Think of a Khan Chacha outlet. Customers (Flutter app) don't walk into the kitchen and cook their own food. They give their order to a waiter (the API). The waiter goes to the kitchen (PostgreSQL), gets what's needed, and brings it back. The customer never sees the kitchen. They don't know the recipes. They definitely don't have keys to the pantry (database password).

The waiter also enforces rules the kitchen doesn't know about — "this table can only order vegetarian", "this customer is banned" — that's your business logic and authentication enforced in the API, not in the database.

REST and HTTP — how the API communicates

Our API uses REST (Representational State Transfer) over HTTP — the same protocol that web browsers use. Flutter sends an HTTP request with a method and a URL. The API responds with JSON data.

Why add an API layer at all?

The app already works. Data persists. CRUD is functional. So why change the architecture? Four reasons — each one worth understanding before we build.

1 — Security: credentials never leave the server

Without API layer (Day 3)

The database password kc_secure_2026 is a constant inside database_service.dart. When you compile the Flutter app into an APK, that constant is embedded in the binary. A motivated attacker can decompile the APK and extract it. Now they have direct access to your PostgreSQL database — read, write, delete, everything.

With API layer (Day 4)

The password lives in a .env file on the Node.js server. Flutter only knows the API's URL — https://api.khanchacha.com/api/requests. No password. No direct database access. Even if someone decompiles the app, they get a URL — not credentials.

2 — Reusability: one API, many apps

Khan Chacha currently has a Flutter mobile app. In the future, they might want:

  • A web dashboard for the head office manager to see all outlets
  • An admin panel for the maintenance team to manage staff accounts
  • An automated WhatsApp notification when a request changes status
  • A monthly PDF report generated by a scheduled script

Without an API: each of these would need direct database access — and its own database credentials. With an API: they all call GET /api/requests or GET /api/dashboard/counts. One source of truth. One set of business rules. Build once, use everywhere.

3 — Business logic lives in one place

Right now, the rule "staff can only see their own outlet's requests" is enforced in Flutter — inside request_list_screen.dart. A clever user could modify the app or call the database directly and bypass this rule.

With an API: the Flutter app sends a request with the user's ID. The API checks the role on the server. If staff, it adds WHERE outlet_id = $user_outlet_id automatically. The user has no way to bypass this — they cannot modify server-side code.

4 — Scalability and cloud migration

Moving the database from localhost to Supabase or AWS RDS is a one-line change in the API's .env file. The Flutter app never needs an update. The API abstracts the database location completely. Tomorrow you could change from PostgreSQL to a different database — and as long as the API returns the same JSON, Flutter wouldn't know or care.

Step 2 — What is Express?

We keep saying "Node.js + Express" but never explaining the difference. Here it is clearly.

Node.js — the engine

Node.js is a JavaScript runtime. It lets you run .js files on a server. Think of it as the engine of a car — it provides the power to run, but it doesn't tell you where to go or how to handle traffic.

Express — the framework on top

Express is a library you install on top of Node.js. It gives you clean tools for defining API routes, handling HTTP requests, sending responses, and wiring middleware. Think of it as the GPS, steering wheel, and dashboard of the car.

Analogy

Node.js alone can handle HTTP — but the code is verbose and repetitive. Without Express, every route would require you to manually parse URLs, read request bodies, set headers, and handle errors from scratch. Express wraps all of that into a clean, readable pattern that the whole industry uses. It is the most downloaded npm package in the world.

Here is what a simple Express route looks like — before any of our Claude Code prompts:

What Express code looks like
const express = require('express');
const app = express();

app.use(express.json()); // parses JSON body from incoming requests

// Define a route: GET /api/requests
app.get('/api/requests', async (req, res) => {
  // req = the incoming request (URL, headers, body, query params)
  // res = the response we send back
  res.json({ message: 'Here are the requests' });
});

app.listen(3000, () => console.log('API running on port 3000'));

That's the mental model — Express lets you define "when someone calls this URL with this method, do this, and send back this response." Claude Code writes all of this for us. But you need to recognise it when you read it.

The .env file — what it does and why it must never be committed

The .env file stores secrets. dotenv reads this file when the server starts and makes each value available via process.env.VARIABLE_NAME. The code never contains the actual secret — only the variable name.

Claude will create a .gitignore that excludes .env and node_modules/ automatically in P1. If you ever accidentally push the .env file, change your database password immediately. GitHub scans public repos for exposed secrets.

The production target — where everything goes in Day 5

The architecture you built today is identical in structure to the production setup. The only differences are where each component runs and how the Flutter app is distributed.

Key insight — Flutter apps are not deployed like websites

A Node.js API is "deployed" to a server — it runs on that server and listens for incoming requests. A Flutter app is not deployed. It is compiled into an installable binary file and distributed to devices. Users download and install it, just like any other mobile or desktop app. The server never runs the Flutter code — it runs on the user's device.

📦 Flutter app — exported, not deployed

You run one build command. Flutter compiles your Dart code into a native binary for the target platform.

Questions & Discussion

Questions from real Day 4 sessions — including errors actually encountered during testing.

Login shows "Invalid credentials" even though the API is running and curl works. Why? ▼

Root cause: auth_service.dart was not updated. AuthService is the login gatekeeper — the LoginScreen calls AuthService().login(username, password), which internally calls another service to do the actual validation. If auth_service.dart still imports database_service.dart and calls DatabaseService().loginUser(), then login will always fail. The postgres package has been removed from pubspec, so DatabaseService can't connect — it throws an exception silently caught by the try/catch, which returns null, which the login screen shows as "Invalid credentials."

Fix: In lib/services/auth_service.dart:
Change import 'database_service.dart'; to import 'api_service.dart';
Change await DatabaseService().loginUser(username, password) to await ApiService().loginUser(username, password)

This is the most common error in Day 4 and is easy to miss because the screens were updated but the auth service wasn't.
Login works now but the dashboard crashes with "type 'Null' is not a subtype of type 'String' in type cast." What's happening? ▼

Root cause: unsafe casting in fromJson(). The API returns JSON where some fields may be null — for example, image_path is optional. If MaintenanceRequest.fromJson() has lines like data['image_path'] as String, Dart throws a null cast exception because you cannot cast null to String.

Fix — apply safe casting to every String field in fromJson():
Instead of: data['field_name'] as String
Use: data['field_name']?.toString() ?? ''

For integers: (data['count'] as num?)?.toInt() ?? 0
For lists: (data['status_history'] as List?)?.map((e) => e.toString()).toList() ?? []

Ask Claude: "Read lib/models/maintenance_request.dart and replace every unsafe 'as String' cast in fromJson() with safe null-tolerant casting using ?.toString() ?? ''"
Which BASE_URL should I use? The docs show 10.0.2.2 but that doesn't work on my Windows machine. ▼
The correct BASE_URL depends entirely on where Flutter is running — not where the API is:
Running Flutter onUse this BASE_URL
Windows desktop (flutter run -d windows)http://localhost:3000/api
Android emulator (flutter run -d emulator)http://10.0.2.2:3000/api
Real Android phone (same WiFi as your machine)http://192.168.x.x:3000/api — get IP with ipconfig
Chrome browser (flutter run -d chrome)http://localhost:3000/api but CORS must be configured in server.js
10.0.2.2 is only meaningful inside the Android emulator — it's the emulator's internal alias for the host machine. On Windows desktop, Flutter runs natively on the same machine as the Node.js server, so localhost works directly.
curl in PowerShell gives "SyntaxError: Expected property name" when I try to test the API. Is the API broken? ▼

The API is not broken. PowerShell's curl is actually an alias for Invoke-WebRequest, not the real curl command. It mangles the JSON body when you escape quotes with backslashes, which sends malformed JSON to the API. The "SyntaxError" is from Node.js trying to parse that malformed JSON.

Use Invoke-RestMethod instead — PowerShell's proper HTTP client: 
Invoke-RestMethod -Uri "http://localhost:3000/api/auth/login" `
  -Method POST `
  -ContentType "application/json" `
  -Body '{"username":"manager","password":"manager123"}'
Single quotes around the JSON body mean PowerShell does not try to interpret its contents. This works reliably on any Windows machine.
Do I need to run both the Flutter app AND the Node.js server at the same time? ▼
Yes — both must be running simultaneously. One terminal runs npm run dev in the API folder (Node.js server). A separate terminal runs flutter run -d windows in the Flutter folder. If the Node.js server is not running, every API call from Flutter will fail with a connection error. In production, the API runs on AWS and is always on — you only need to start it manually during local development.
Can I test the API without Flutter running? ▼
Yes — and you should. Use curl in the terminal, or install a GUI tool like Postman or Insomnia. Testing the API independently confirms it works before adding the Flutter layer. If the API test fails, fix the API. If the API test passes but Flutter fails, the issue is in ApiService. Separating these two helps you debug faster.
The API returns data but Flutter shows an empty list. What's wrong? ▼
Three likely causes: (1) The JSON field names from the API don't match what MaintenanceRequest.fromJson() expects — check snake_case vs camelCase. (2) The Flutter screen is showing an empty list because setState() wasn't called after the data loaded — check the loading logic. (3) The BASE_URL in ApiService is wrong — log the URL and the raw response body to confirm what Flutter is actually receiving. Ask Claude: "My API returns correct JSON via curl, but Flutter shows an empty list. Read ApiService and the relevant screen and find the mismatch."
Can other students' phones also connect to my Node.js API? ▼
Yes, if they're on the same WiFi network. Find your machine's local IP: ipconfig on Windows, ifconfig on Mac/Linux. Share that IP with other students. They change their ApiService BASE_URL to http://192.168.x.x:3000/api. As long as both machines are on the same router and your firewall doesn't block port 3000 — it works. This is a great live demo: one student creates a request on their phone, another student's app pulls to refresh and sees it appear.
What happens to our direct DatabaseService — should we delete it? ▼
Keep it for now — don't delete it. Once the API is confirmed fully working, you can remove it or keep it as a reference. Some teams keep both and use a feature flag to switch between them for testing. For this course, once P12 is complete and everything tests green, you can safely remove database_service.dart and the postgres package from pubspec.yaml.
Is this API secure enough for production? ▼
Not yet — and that's intentional for Day 4. Three gaps to fix before production: (1) Passwords are compared in plain text — add bcrypt to hash them. (2) There's no JWT token — any caller can hit any endpoint without proving who they are. Day 5 adds JWT authentication. (3) Role enforcement happens in Flutter — it needs to move to the API (server-side). These are Day 5 topics. Day 4's job is to establish the API layer correctly so Day 5 can add security on top of it.
