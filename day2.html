🏗️App Architecture — Folder Structure & Data Flow

Before writing a single prompt, understand how the app is organized. This is how you tell Claude where to put things.

The Three Layers Rule

Every well-structured Flutter app has three layers. When you prompt Claude, always tell it which layer you're working in.

📦 Models

Pure data. No UI. Just the shape of your data — what fields a MaintenanceRequest has. Never touches the screen.

🖥️ Screens

Full pages. HomeScreen is the list. DetailScreen shows one request. Each screen uses widgets to build its layout.

🧩 Widgets

Reusable pieces. RequestCard is used inside HomeScreen. The form is a widget opened from the FAB. Build once, use anywhere.

🎨Widgets & State — The Building Blocks

Every piece of UI in Flutter is a Widget. Understanding two types — Stateless and Stateful — helps you prompt Claude correctly.

Everything is a Widget

In Flutter, everything visible on screen is a Widget. Text is a widget. A button is a widget. A row of items is a widget. The whole screen is a widget. Widgets nest inside each other to build complex UIs.

StatelessWidget

Never changes after it's built. What you pass in, stays in. Good for:

  • Request cards (display only)
  • Status badges
  • Header text
  • Icons
class StatusBadge extends StatelessWidget {
  final String status;
  const StatusBadge({required this.status});

  @override
  Widget build(context) {
    // Returns the coloured pill
  }
}

StatefulWidget

Can change while the app runs. Triggers a re-draw when data changes. Good for:

  • HomeScreen (filter tab changes)
  • New request form (typing, validation)
  • Any list that updates
class HomeScreen extends StatefulWidget {
  // selectedFilter changes when user
  // taps All / Open / Pending / Closed
  String selectedFilter = 'All';

  void updateFilter(String f) {
    setState(() => selectedFilter = f);
  }
}

setState() — How Flutter Knows to Redraw

When the user taps a filter tab, setState() is called. Flutter re-runs the build() method of that widget, and only that widget updates on screen. This is called reactive UI — the screen reacts to state changes.

Navigator — How Screens Connect

When a user taps a request card, we "push" a new screen on top. When they press Back, we "pop" it.

// Back button automatically calls:
Navigator.pop(context);

💬Prompting Patterns — The Art of Talking to Claude

Your code quality is only as good as your prompt quality. These patterns separate prompts that work from prompts that waste time.

The Anatomy of a Good Prompt

A good Claude Code prompt has four parts:


[CONTEXT]  — What app is this? What already exists?
[TASK]    — Exactly what do you want to add/change?
[RULES]   — Constraints (files to touch, style to keep, fields to include)
[OUTPUT]  — What should the result look like / behave like?

Pattern 1 — The Reference Prompt

Tell Claude which file to look at. Never assume it knows what exists.

In our Khan Chacha Flutter app, look at lib/widgets/request_card.dart. The card currently shows productName, category, outletId, and requestDate in a horizontal row. I want to also show a small icon next to the category label (⚡ for Electrical, ❄️ for AC, 🍳 for Kitchen). Only modify request_card.dart. Do not change any other file.

Pattern 2 — The Constraint Prompt

Tell Claude what NOT to change. This prevents it from refactoring code you didn't ask it to touch.

In lib/screens/home_screen.dart, add a search bar at the top of the list that filters by productName. IMPORTANT constraints: - Do not change the filter tab row (All / Open / Pending / Closed) — keep it exactly as is - Do not change the RequestCard widget - The search should work together with the tab filter (both active at the same time) - Keep the existing state management pattern (setState)

Pattern 3 — The Context Dump Prompt

For complex tasks, paste the existing code into the prompt. Claude works with exactly what you have, not what it imagines.

Here is our current MaintenanceRequest model: [paste full maintenance_request.dart here] I need to add a fromJson() and toJson() method so this model can be serialized to and from JSON (needed to store and retrieve data from an API tomorrow). Map every field. Use String types throughout — no DateTime, no enums. Return the updated file.

Pattern 4 — The Outcome Prompt

Describe what the user experiences, not what widgets to use. Claude picks the right widget.

❌ Vague / Widget-Focused

"Add a Container with a Row inside with two Text widgets and a Chip widget for the category."

You're telling Claude how, not what. You might be wrong about the best widget choice.

✅ Clear / Outcome-Focused

"Each request card should show the product name in bold, and below it a small coloured tag showing the category. The tag colours are: purple for Electrical, blue for AC, orange for Kitchen."

Claude chooses the right widgets. You describe the result.

Pattern 5 — The Step-by-Step Breakdown

For features that touch multiple files, break them into steps. Ask Claude to do one step at a time and confirm before moving to the next.

I want to add a Dashboard screen to the Khan Chacha app. Let's do this in 3 steps. For now, only do Step 1: STEP 1: Create lib/screens/dashboard_screen.dart with a basic Scaffold that shows the text "Dashboard — Coming Soon" in the center. Add a route for this screen in main.dart. Add a bottom navigation bar to HomeScreen with two tabs: "Requests" and "Dashboard". Stop after Step 1. I will test and then ask you for Step 2.

Pattern 6 — The Fix Prompt

When something breaks, give Claude the error message — never just say "it's not working."

My Flutter app is showing this error in the terminal: "The named parameter 'category' is required, but there's no corresponding argument." This happened after I added the category field to MaintenanceRequest. The error is in lib/data/sample_requests.dart. Look at the constructor in maintenance_request.dart and fix sample_requests.dart so that all sample records include the category field with a valid value (Electrical, AC, or Kitchen).

❌ Level 1"Make it look better"Random changes, probably not what you wanted
⚠️ Level 2"Add a search bar to the home screen"Works, but may not match your existing style
✅ Level 3"Add a search bar below the AppBar in home_screen.dart that filters the request list by productName. Keep the existing tab filter."Exactly what you wanted, in the right place

🗄️Friday: PostgreSQL + Dashboards (Permanent Data)

The step from Google Sheets to a real database. Everything your students learned in PostgreSQL training comes alive here.

Why Move from Sheets to PostgreSQL?

Google Sheets — Good for now

  • Easy to set up
  • Non-technical managers can edit it
  • But: no real-time updates
  • But: no access control per row
  • But: 10M cell limit
  • But: no joins, foreign keys, or queries

PostgreSQL — Production ready

  • Real transactions (no duplicate requests)
  • Row-level security (managers see only their outlet)
  • Real-time subscriptions (auto-update the list)
  • Proper relationships (outlets table, assets table)
  • SQL queries for dashboard analytics

The Tool: Supabase

Supabase is PostgreSQL + REST API + Real-time + Authentication — all in one hosted service. Students who know PostgreSQL from their training can write raw SQL queries in Supabase's dashboard. The Flutter SDK talks to Supabase exactly like it would talk to any REST API.

Think of Supabase as: PostgreSQL that has an API already built for it. No Express server. No manual REST endpoints. Write a table, get an API. Write a SQL query, get a REST endpoint.

-- PostgreSQL table for maintenance requests
CREATE TABLE maintenance_requests (
  id              TEXT PRIMARY KEY DEFAULT gen_random_uuid()::TEXT,
  product_name    TEXT NOT NULL,
  product_id      TEXT NOT NULL,
  category        TEXT CHECK (category IN ('Electrical', 'AC', 'Kitchen')),
  status          TEXT CHECK (status IN ('Open', 'Pending', 'Closed')) DEFAULT 'Open',
  request_date    DATE NOT NULL DEFAULT CURRENT_DATE,
  description     TEXT,
  manager_name    TEXT NOT NULL,
  outlet_id       TEXT NOT NULL,
  outlet_name     TEXT NOT NULL,
  outlet_address  TEXT,
  created_at      TIMESTAMPTZ DEFAULT NOW()
);

-- Dashboard query: requests by status
SELECT status, COUNT(*) AS total
FROM maintenance_requests
GROUP BY status;

-- Dashboard query: busiest outlets
SELECT outlet_name, COUNT(*) AS request_count
FROM maintenance_requests
GROUP BY outlet_name
ORDER BY request_count DESC;

Friday's database_service.dart (Preview)

// lib/services/database_service.dart
import 'package:supabase_flutter/supabase_flutter.dart';
import '../models/maintenance_request.dart';

class DatabaseService {
  final _client = Supabase.instance.client;

  Future<List<MaintenanceRequest>> fetchAll() async {
    final data = await _client
        .from('maintenance_requests')
        .select()
        .order('request_date', ascending: false);
    return (data as List).map((row) =>
        MaintenanceRequest.fromJson(row)).toList();
  }

  Future<void> updateStatus(String id, String newStatus) async {
    await _client
        .from('maintenance_requests')
        .update({'status': newStatus})
        .eq('id', id);
  }
}

❓Q&A — Questions, Discussion & Common Issues

Common questions from Day 2 sessions, and how to think about them.

FAQ — Day 2

Q: Claude changed something I didn't ask it to change. What do I do?

This is the most common issue. You have two options:
1. Use Ctrl+Z to undo and re-prompt with an added constraint: "Only modify [filename]. Do not change any other file."
2. Ask Claude to revert the unwanted change: "In your last response, you also changed [widget]. Revert that file to its previous state."

Q: My app compiles but the screen is blank / shows an error widget. How do I debug?

Look at the Flutter terminal. There will be a red error with a file name and line number. Copy the full error and give it to Claude: "After adding [feature], my app shows this error: [paste error]. Fix it without changing anything else."

Q: I don't understand what the code does. Should I ask Claude to explain it?

Yes — and you should do this regularly. After Claude generates code, ask: "Explain what the setState() call on line 34 does in plain English, without using coding jargon." Understanding why the code works (even loosely) makes you a better prompter.

Q: Can I build a real app this way without knowing Dart deeply?

Yes — and this training is proof. You need to understand the outcome (what the user experiences), the structure (which file to change), and how to write a good prompt. Claude handles the Dart. The skill you're building is: describing software precisely. That's a skill that transfers to every tech role you'll ever have.

Q: Claude gave me code that works, but I ran it again and got different code. Why?

Claude is a language model — it doesn't remember your previous prompts in a new session. Always save your working code (commit to GitHub) before running a new prompt. If Claude produces a worse solution second time, you can revert to your saved version.

🔥 The Most Powerful Habit to Build

Read Before You Run

Before you accept Claude's code and press hot reload, spend 60 seconds reading through what it produced. You don't need to understand every line. You're checking:

  • Did it touch the right files?
  • Does the structure look similar to existing code (same naming style, same widget pattern)?
  • Does the logic match what I asked for at a high level?

If something looks wildly different from what you expected — ask Claude to explain it before running it. This prevents bugs that are hard to trace back.
