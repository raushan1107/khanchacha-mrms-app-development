What exactly is PostgreSQL?

PostgreSQL is a free, open-source database management system. It stores your data in structured tables — like a highly reliable, queryable filing cabinet — and lets multiple users read and write to it simultaneously from anywhere on the network.

It is often called "Postgres" for short. The full name is PostgreSQL (pronounced "post-gress-Q-L"). It is the most feature-complete open-source relational database available today and has been in active development for over 35 years.

In one sentence

PostgreSQL is a program that runs on a computer, listens for incoming data requests, stores data safely on disk, and answers questions about that data using the SQL language.

A brief history — why this matters

PostgreSQL started in 1986 at the University of California, Berkeley. The original project was called POSTGRES (Post-INGRES — a successor to the Ingres database). It became open-source in 1994 and has been developed by a global community of volunteers and companies ever since.

Why the age matters

35+ years of production use means every edge case, every hardware failure, every weird bug has already been found and fixed by someone else. When you store Khan Chacha's maintenance data in PostgreSQL, it is guarded by three decades of battle-testing.

Why open-source matters

No licence fees. No vendor lock-in. You can run it on your laptop today, on a company server tomorrow, on AWS or Azure next month — using the exact same SQL. Nobody can charge you to keep using it.

What does "relational" mean?

Relational means your data is stored in tables (like spreadsheet sheets), and those tables can be linked to each other using shared identifiers.

Analogy

Think of a franchise operation like Khan Chacha. You have three filing cabinets: one for outlets, one for employees, one for maintenance requests. Every maintenance request has a slip inside it that says "this was raised by Employee 14 from Outlet 3." You don't copy the employee's full name and outlet address onto every single request — you just write the reference numbers, and you go check the other cabinet when you need the full details. That's exactly what a relational database does. The reference numbers are called foreign keys.

The alternative is a "flat file" — storing everything in one big table with columns repeated everywhere. That leads to inconsistency (the same outlet name spelled differently in different rows) and makes updates a nightmare (change an outlet's address and you have to update 500 rows).

ACID — in plain English

PostgreSQL guarantees ACID properties. This is why banks, hospitals, and governments trust it with critical data.

Why not other databases?

Every database exists for a reason. The right choice depends on your data's shape, your team's size, and your scale. Here's why each alternative falls short for Khan Chacha — and why PostgreSQL wins.

SQLite — too basic for multi-user apps

Where SQLite is great

  • Mobile apps storing user preferences (it's built into every Android and iOS device)
  • Desktop apps with a single user
  • Prototypes and tests that run fast
  • Embedded systems (your browser's storage is SQLite)

Why not for Khan Chacha

  • No server — it's a file, not a service. Only one device can write at a time.
  • No concurrent users — Outlet A and Outlet B cannot share the same SQLite file
  • No network access — can't be reached from another phone
  • No user management — no roles, no permissions

MySQL — a valid competitor, but PostgreSQL goes further

Where MySQL is used

  • WordPress, Joomla, Drupal (the LAMP stack)
  • Read-heavy web applications
  • Legacy systems from the early 2000s

Why PostgreSQL wins

  • Better standards compliance (ANSI SQL)
  • Richer data types (JSONB, arrays, UUID, custom types)
  • Stronger transaction guarantees
  • More advanced query planner — GROUP BY + window functions for dashboard analytics
  • Supabase (our cloud path) is built on PostgreSQL, not MySQL

MongoDB — wrong shape for maintenance data

Where MongoDB excels

  • Data with unpredictable or varying structure (social media posts, event logs)
  • Rapid prototyping when schema isn't clear yet
  • Content management where each document is different

Why not for Khan Chacha

  • Our data is highly structured — every request has exactly 13 known fields
  • We have clear relationships: requests → outlets → users. MongoDB doesn't enforce these.
  • Dashboard queries (GROUP BY status, COUNT per outlet) are awkward in MongoDB's aggregation pipeline. They're 2-line SQL.
  • ACID transactions across collections are complex and slow

Firebase Firestore — too expensive, too locked-in

Where Firebase works

  • Real-time chat apps and collaborative tools
  • Fast prototyping with Google auth built-in
  • Small-to-medium apps that want zero backend setup

Why not for Khan Chacha

  • Google-owned and priced per read/write — costs scale sharply with usage
  • No SQL — you cannot write SELECT status, COUNT(*) GROUP BY status
  • Schema migrations are a pain — no ALTER TABLE equivalent
  • Vendor lock-in: if Google discontinues or reprices, you're stuck
  • On-premises is impossible — Firebase is cloud-only

Why PostgreSQL is the right choice for MRMS

The winning combination

  • Structured, fixed schema: 13 fields per request, always. PostgreSQL enforces this — no missing fields, no wrong types.
  • Multi-outlet, multi-user: 20+ outlets, all hitting the same database concurrently. PostgreSQL handles thousands of simultaneous connections.
  • Dashboard analytics: SELECT status, COUNT(*) FROM maintenance_requests GROUP BY status — two lines of SQL, instant result. That's the "requests by status" pie chart done.
  • On-premises today: Runs on a laptop right now. No cloud account needed, no internet dependency, no per-request billing.
  • Cloud-ready tomorrow: When ready to scale, the same database moves to Supabase (PostgreSQL in the cloud) by changing three config values. The SQL stays identical.
  • Open source forever: Khan Chacha will never be billed for using it, and the code is auditable.

Installing PostgreSQL on your machine

Choose your operating system. After installation, all three platforms have the same PostgreSQL — the commands and SQL are identical regardless of OS.

🪟 Windows 🍎 macOS 🐧 Linux

Windows — Graphical Installer (recommended for beginners)

1
Download the installer
Go to postgresql.org/download/windows and click the "Download the installer" link for the latest version (16.x). This is a ~300 MB installer from EDB (EnterpriseDB — the company that maintains the Windows build).
2
Run the installer
Accept defaults for installation directory and components. Make sure pgAdmin 4 and Command Line Tools are checked. When it asks for a password — this is the password for the postgres superuser account. Write it down. You will need it today.
3
Set the port and locale
Default port is 5432 — leave it as is. Locale: accept the default. Click through the rest of the wizard and let it install. Takes 2–3 minutes.
4
Skip the "Stack Builder" at the end
The installer will offer to open Stack Builder (to install extensions). Uncheck it and click Finish. You don't need it for now.
5
Add PostgreSQL to your PATH
Open Start → search "Environment Variables" → Edit the system environment variables → Environment Variables → find "Path" in System variables → Edit → Add: C:\Program Files\PostgreSQL\16\bin (adjust version number if different). This lets you type psql in any terminal.

macOS — Two options

Option A — Postgres.app (simplest, recommended)

Go to postgresapp.com, download the app, drag it to Applications, and open it. Click "Initialize" to start a fresh PostgreSQL server. It runs in your menu bar. Double-click a database name to open a terminal connected to it. Done.

Then add the CLI tools to your path — the app's website has a one-line command to copy-paste.

Option B — Homebrew

brew install postgresql@16
brew services start postgresql@16
echo 'export PATH="/opt/homebrew/opt/postgresql@16/bin:$PATH"' >> ~/.zshrc
source ~/.zshrc

Create the Khan Chacha database

In the psql prompt, run these commands to create a dedicated database and a user for the app:

-- Create the database
CREATE DATABASE khan_chacha;

-- Create an application user (better practice than using the superuser)
CREATE USER kc_app WITH PASSWORD 'kc_secure_2026';

-- Grant this user full access to our database
GRANT ALL PRIVILEGES ON DATABASE khan_chacha TO kc_app;

-- Connect to the new database
\c khan_chacha

-- Grant schema permissions
GRANT ALL ON SCHEMA public TO kc_app;
Verify it worked

Type \l — you should see khan_chacha in the list of databases with kc_app as the owner.

Claude is your SQL pair-programmer

Just as we used Claude Code to write Flutter and Dart without memorising every widget, we use Claude to write PostgreSQL without memorising every SQL syntax rule. The goal is the same: describe what you want in plain English, Claude writes it, you review, paste, and verify.

What changes — and what doesn't

You still need to understand what you are building — how many tables, what the relationships are, why an index matters. Claude handles the exact syntax. The understanding of the data model is yours. The typing is Claude's. This is the same division of labour we established on Day 1 with Flutter.

One important rule — always read before you run

In Flutter, running a bad prompt rebuilds a screen incorrectly — easy to undo. In SQL, a DROP TABLE or a DELETE without a WHERE clause destroys data permanently. Before you paste any SQL Claude generates into pgAdmin or psql, read it. Make sure it says what you expect. This takes 30 seconds and saves hours of recovery.

Prompt 5 — Alter the schema as the app grows

The Khan Chacha schema will evolve. When you need to add a column, rename a field, or add a new table, never re-create everything from scratch. Use ALTER TABLE. Ask Claude to write it.

Thinking in tables before writing SQL

👈 Don't want to write this SQL yourself?

Before you write a single CREATE TABLE, list the entities in your system.

  • Outlets — each Khan Chacha branch. Has an ID, name, and address. Doesn't change often.
  • Users — managers and staff. Each user belongs to one outlet. Has credentials and a role.
  • Maintenance Requests — the core of the app. Each request is raised by one user at one outlet. This is where most reads and writes happen.

The relationships: a request belongs to an outlet. A user belongs to an outlet. A request is raised by a user. This is how we avoid repeating the outlet name and address in every single request row.

Design rule — never store what you can look up

The maintenance_requests table stores the outlet_id. It does not store outlet_name or outlet_address — those live in the outlets table. When you need the address, JOIN the two tables. This means updating an outlet's address is one row change, not a bulk update across all 500 requests.

SQL
CREATE TYPE request_status AS ENUM ('Open', 'Pending', 'Closed'); CREATE TYPE request_category AS ENUM ('Electrical', 'AC', 'Kitchen'); CREATE TABLE maintenance_requests ( id TEXT PRIMARY KEY, product_name TEXT NOT NULL, product_id TEXT NOT NULL, category request_category NOT NULL, status request_status NOT NULL DEFAULT 'Open', request_date DATE NOT NULL DEFAULT CURRENT_DATE, description TEXT NOT NULL, image_path TEXT, -- nullable: photo evidence outlet_id TEXT NOT NULL REFERENCES outlets(id), raised_by TEXT NOT NULL REFERENCES users(id), status_history JSONB NOT NULL DEFAULT '[]', -- array of change logs created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW() ); -- Index for the most common filter: status CREATE INDEX idx_requests_status ON maintenance_requests(status); -- Index for outlet-level filtering (staff view) CREATE INDEX idx_requests_outlet ON maintenance_requests(outlet_id); -- Index for date-based sorting CREATE INDEX idx_requests_date ON maintenance_requests(request_date DESC);

Why JSONB for status_history?

Instead of a separate status_change_log table (which adds complexity), we store the history as a JSON array in a JSONB column. Each entry looks like {"from":"Open","to":"Pending","at":"2026-07-06T14:30:00"}. PostgreSQL can query inside JSONB natively. It's the pragmatic choice for this use case — simple to read from Flutter, no extra JOIN needed.

Basic SELECT — reading rows

Run in pgAdmin or DBeaver query editor
-- Read all columns from all rows SELECT * FROM maintenance_requests; -- Read only specific columns SELECT id, product_name, status, request_date FROM maintenance_requests;

The * means "all columns." You'll rarely use it in production (fetching unused columns wastes bandwidth), but it's perfect for exploring data.

Filtering with WHERE

SQL
-- All Open requests SELECT id, product_name, outlet_id, request_date FROM maintenance_requests WHERE status = 'Open'; -- Open requests from Outlet OUT-02 only (staff view) SELECT id, product_name, category, request_date FROM maintenance_requests WHERE status = 'Open' AND outlet_id = 'OUT-02'; -- All AC or Kitchen requests SELECT id, product_name, category FROM maintenance_requests WHERE category IN ('AC', 'Kitchen');

Sorting with ORDER BY

SQL
-- Newest requests first (this is the default sort in the Flutter list) SELECT id, product_name, status, request_date FROM maintenance_requests ORDER BY request_date DESC; -- Oldest first SELECT id, product_name, status, request_date FROM maintenance_requests ORDER BY request_date ASC; -- By outlet, then by date within each outlet SELECT id, product_name, outlet_id, request_date FROM maintenance_requests ORDER BY outlet_id, request_date DESC;

Aggregates — COUNT, GROUP BY

This is where SQL starts to feel powerful. Aggregates answer "how many?" questions — exactly what the dashboard needs.

SQL
-- Total number of requests SELECT COUNT(*) FROM maintenance_requests; -- How many requests per status → this powers the dashboard pie chart SELECT status, COUNT(*) AS total FROM maintenance_requests GROUP BY status; -- Result: Open|4 Pending|2 Closed|2 -- How many requests per outlet → busiest outlets chart SELECT outlet_id, COUNT(*) AS request_count FROM maintenance_requests GROUP BY outlet_id ORDER BY request_count DESC; -- Category breakdown for the past 30 days SELECT category, COUNT(*) AS total FROM maintenance_requests WHERE request_date >= CURRENT_DATE - INTERVAL '30 days' GROUP BY category ORDER BY total DESC;

JOINs — combining tables

JOINs are how you bring data from two tables together. Instead of storing the outlet name on every request, we JOIN to the outlets table when we need it.

SQL
-- Get requests with full outlet name and address (for the detail screen) SELECT r.id, r.product_name, r.category, r.status, r.description, o.name AS outlet_name, o.address AS outlet_address, u.name AS manager_name FROM maintenance_requests r JOIN outlets o ON r.outlet_id = o.id JOIN users u ON r.raised_by = u.id WHERE r.status = 'Open' ORDER BY r.request_date DESC;

Is SQL easy or tough? An honest answer.

Honest Assessment

Basic SQL is easy. SELECT, WHERE, ORDER BY, and even GROUP BY are readable in plain English. You can write all three dashboard queries above within your first hour of learning SQL. The logic maps directly to plain-language questions: "Show me all open requests, newest first" translates almost word for word into SQL.

Advanced SQL takes time. Complex JOINs across many tables, window functions, recursive queries, query optimization — these take months to get comfortable with. You won't need them for the Khan Chacha app.

The hard part isn't the syntax. It's thinking in sets. SQL operates on entire tables at once, not row by row. Once that mental model clicks, everything else follows. It usually clicks after writing about 20–30 real queries on real data you care about.

On-premises — what it means and why we start here

On-premises means the database runs on a machine that you own and control — your laptop today, a company server later. There is no third-party cloud provider involved. You are the data centre.

Why on-premises is right for Days 3–4

  • Zero cost — no cloud bills, no sign-up, no credit card
  • Works without internet — train, office without stable WiFi, anywhere
  • You see and control everything — great for learning
  • The SQL, schema, and Flutter code you write today is 100% identical to what you'd run on the cloud

The honest limitations of on-premises

Why you can't ship to production with a laptop database

  • Always-on requirement: The database only works while the machine running it is switched on and connected. A laptop closes, the app for all 20 outlets breaks.
  • Network access: The Flutter app connects to localhost. No other phone on another WiFi network can reach it. Outlet managers across Delhi cannot use the same database that's on your laptop.
  • Backups: You are responsible for backing up the database. If the hard drive fails, all data is gone.
  • Server room cost: A dedicated on-premises server for Khan Chacha means a machine that runs 24/7 in a server room with power backup (UPS), physical security, IT maintenance, and cooling. That's expensive for a restaurant chain.
  • Updates and patches: You manually apply PostgreSQL security updates and OS patches. Cloud providers do this automatically.

Questions & Discussion

Common questions from this session, answered.

Do I need to keep PostgreSQL running all the time while building the app? ▼
Yes — while developing locally, PostgreSQL must be running on your machine for the app to connect. On Windows, it starts automatically as a service when your PC boots (you can verify in Services → postgresql-16). On Mac with Postgres.app, click the elephant icon in the menu bar. On Linux, sudo systemctl start postgresql starts it and sudo systemctl enable postgresql makes it start automatically on boot. You only notice this matters when you try to run the app and the DB is off — it shows a connection error.
Can another phone on my WiFi network connect to my local PostgreSQL? ▼
Yes, with configuration changes — but we don't recommend it for this training. By default, PostgreSQL only accepts connections from localhost (the same machine). To allow other machines on the local network: edit pg_hba.conf to allow connections from your local IP range, edit postgresql.conf to set listen_addresses = '*', restart PostgreSQL, and update the Flutter app's host to your machine's local IP (e.g. 192.168.1.5) instead of localhost. This works fine for demo on a local network — but it's a lab setup, not production.
What happens to the data if I reinstall PostgreSQL? ▼
Reinstalling can erase all data. Always take a backup first: pg_dump -U postgres khan_chacha > khan_chacha_backup.sql. This saves all your tables and data as a SQL file. Restore with: psql -U postgres khan_chacha < khan_chacha_backup.sql. Make this a habit — run a backup at the end of each training day.
We have 5 students, each with their own laptop. How do we share the same database? ▼
For this training: each student runs their own PostgreSQL locally. Everyone has the same schema and same sample data — it's like each person having their own copy of the whiteboard to practice on. When you move to Supabase (the cloud path), everyone's Flutter app points to the same Supabase project URL. Then all five students are reading and writing to the same database, exactly like the real production scenario.
Are we storing passwords as plain text in the database? Is that safe? ▼
For this training: yes, passwords are plain text in the users table. This is intentional — it's clearer for learning. In production, you never store plain passwords. You use a hashing algorithm (bcrypt, argon2, or PBKDF2) that converts "manager123" into a long random-looking string. When the user logs in, you hash what they typed and compare the hash — you never compare raw passwords. When we move to Supabase, their built-in Auth handles all of this for us.
Why don't we just use Supabase from day one? ▼
Three reasons: First, Supabase abstracts away the database — you click a UI and tables appear. That's great for speed but bad for understanding. Writing raw SQL, seeing the data in pgAdmin, understanding JOINs and GROUP BY — that's the learning. Second, Supabase requires a stable internet connection. A training session on local PostgreSQL works in a hotel room with bad WiFi. Third, when something breaks in production (and it will), knowing how PostgreSQL actually works is what lets you debug it — whether it's running locally or on Supabase.
The Flutter app gets a "Connection refused" error. What do I check? ▼
Check in this order:
  1. Is PostgreSQL running? Run psql -U postgres -c "SELECT 1;" — if that fails, start the PostgreSQL service.
  2. Is the port correct? Default is 5432. Run SELECT * FROM pg_settings WHERE name='port'; in psql to confirm.
  3. Are the credentials correct? Try logging in with psql -h localhost -U kc_app -d khan_chacha. If that fails, the username/password/database name in your Dart code won't work either.
  4. Is SslMode.disable set? Some configurations reject non-SSL connections. Conversely, some require SSL. Match your pg_hba.conf setting.
  5. On Android emulator: the emulator's localhost is 10.0.2.2, not 127.0.0.1. If running on emulator, change the host in DatabaseService to '10.0.2.2'.
