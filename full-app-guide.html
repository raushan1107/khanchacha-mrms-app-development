🎯 Context — What We Are Building & Why

Before writing a single prompt, understand the full picture. The clearer your mental model, the better your prompts will be.

The Khan Chacha Problem

Khan Chacha runs multiple restaurant outlets across Delhi/NCR. When a bulb fuses, an AC breaks, or a microwave fails — the outlet manager currently fills a Google Form. That form saves to a spreadsheet. Someone eventually reads it. There is no real-time tracking, no approval chain, no image evidence, no history. We are replacing that process with a proper mobile app.

The 5 Screens We Will Build

Screen 1 — Login

Split layout. Left: Khan Chacha brand panel (dark, kitchen motif). Right: Clean login form. Two dummy accounts — Manager and Staff. Animated transition into the app on successful login.

Screen 2 — Dashboard

Role-aware landing page. Manager sees summary counters (Open/Pending/Closed) and approval queue. Staff sees a "Raise Request" CTA and their own submitted requests only.

Screen 3 — Request List

Filterable, searchable list of all maintenance requests (Manager) or only own requests (Staff). Filter tabs + category chips + sort. Compact horizontal cards.

Screen 4 — Request Detail

All 13 fields (12 + image). Action buttons: Update Status, Edit (with confirmation dialog), Delete (with confirmation dialog). Image displayed if uploaded.

Screen 5 — New / Edit Form

Full form for creating or editing a request. Pre-filled when editing. Image picker button. Category dropdown (Electrical/AC/Kitchen). All validation. Confirmation before updating existing records.

The Art of Prompting — Recap

Rule 1 — Describe outcomes, not widgets

Don't say "add a Container with a Row". Say "each card shows the product name bold on the left and the status badge on the right." Claude picks the right widget.

Rule 2 — Always name the file

Claude works better when you say "in lib/screens/detail_screen.dart" than when you say "in the detail page". Be precise.

Rule 3 — Use CONSTRAINTS to protect what exists

Add "Do not change any other file" or "Keep the existing card style unchanged" at the end. Prevents Claude from refactoring code you didn't ask it to touch.

Rule 4 — The capstone prompt wins because it has full context

A big prompt isn't better because it's long. It's better because it gives Claude the complete picture — what the app is, who uses it, what every screen must do, and how they connect. Claude produces better code when it understands the product.

Two Ways to Build the Same App

🔨 Progressive (Step-by-Step)

10 prompts. You run one at a time, test the app, then move to the next. You see the app grow screen by screen. Best for learning how it all fits together.

🚀 Capstone (One Big Prompt)

One prompt. Run it in a brand-new folder. Claude builds the entire app in one shot. Best for seeing what a production-quality prompt looks like.

⚠️ Important: Step-by-Step and Capstone are two different Flutter projects in two different folders. Do not mix them. The step-by-step continues your existing khan_chacha_mrms folder. The capstone creates a brand-new project.

Authentication — How Login Works

lib/services/auth_service.dart
Singleton service. One instance exists for the entire app session.
  • How the singleton works: factory AuthService() => _instance; — every time you write AuthService() anywhere in the app, you get the same object.
  • login(username, password): Searches dummyUsers list. Compares both username and password as plain strings. Returns the matching UserAccount or null if no match. Sets currentUser on success.
  • currentUser: Holds the logged-in user object. Passed to every screen that needs to know who is using the app. Screens check currentUser!.role to decide what to show.
  • logout(): Sets currentUser = null. The screen then uses Navigator.pushAndRemoveUntil to clear the navigation stack and return to Login so Back button doesn't bypass logout.
  • Where validation happens: Entirely in auth_service.dart, called from login_screen.dart. No validation logic lives in the UI file itself.

File Responsibility Map

lib/main.dart
App entry point. Defines MaterialApp, global theme (amber/navy), and sets LoginScreen as the home screen. No business logic here.
lib/models/maintenance_request.dart
The data shape for a single request. Pure Dart — no UI, no imports of Flutter widgets.
  • copyWith(): Returns a new MaintenanceRequest with some fields replaced. Used when updating status or editing — we never mutate the original object directly.
  • fromJson() / toJson(): Converts between the object and a Map. Used when reading from / writing to Google Sheets or a database (future sessions).
lib/models/user_account.dart
Shape of a user (manager or staff). Holds credentials, name, role, and their assigned outlet. Used by AuthService and passed to every screen.
lib/data/dummy_data.dart
All in-memory data for the prototype.
  • dummyUsers: Fixed list. Do not modify during runtime — only used for auth lookup.
  • sampleRequests: Mutable list. When a request is created, edited, or deleted, the app modifies this list directly. Since it's in memory, changes are lost when the app restarts — that's intentional for the prototype.
lib/services/navigation_service.dart
All page route animations defined in one place. No animation code lives inside screen files.
  • slideFromRight: Standard forward navigation (list → detail, dashboard → list)
  • slideFromBottom: Form opening (feels like a bottom sheet but is a full screen)
  • fade: Login → Dashboard and Logout → Login
lib/screens/login_screen.dart
Split-panel login UI. Left panel is purely decorative (StatelessWidget). Right panel is a StatefulWidget managing username, password, isLoading state.
  • Validates inputs locally (not empty checks only)
  • Calls AuthService().login() for credential matching
  • Shows CircularProgressIndicator for 800ms to simulate auth latency
  • On success: fades to DashboardScreen passing the UserAccount object
lib/screens/dashboard_screen.dart
Role-aware landing page after login. Decides what to show based on currentUser.role.
  • Stat counts are computed by filtering sampleRequests by status (and outletId for staff)
  • Stat cards animate in with staggered AnimatedOpacity on initState
  • Bottom nav manages which body (home vs list) is shown using a selectedIndex integer in state
lib/screens/request_list_screen.dart
The filterable/searchable request list. All four filters run in sequence inside a computed getter.
  • Filter pipeline: role filter → status tab → category chip → search text → sort
  • Dismissible (Manager only): swipe to delete with confirmation dialog
  • RefreshIndicator: simulates a reload with 800ms delay
  • Each card uses Navigator.push + await so the list can update if a request changes in the detail screen
lib/screens/detail_screen.dart
Full view of one request. Also manages status updates, edit, and delete actions for the Manager role.
  • Status update: uses copyWith(status: newStatus) then replaces the record in sampleRequests and calls setState()
  • Returns updated record via Navigator.pop(context, updatedRequest) — list screen awaits this
  • Delete: removes from sampleRequests by id, then pops
  • Hero tag on the top banner: animates the gradient container from the card's category colour
lib/screens/request_form_screen.dart
Handles both Create and Edit mode. Receives existingRequest as an optional parameter. If null → Create Mode.
  • In Edit Mode: TextEditingControllers are pre-populated from existingRequest fields in initState()
  • Outlet ID, Outlet Name, Manager Name are always pre-filled from currentUser and rendered as read-only (grey background)
  • Image picker: ImagePicker().pickImage(source: ImageSource.gallery) returns XFile?. The path (xFile.path) is stored as a String in state and passed into the request model.
  • On submit: Create mode generates the next REQ-00X id. Edit mode uses copyWith() preserving the original id and statusHistory.
lib/widgets/request_card.dart
Compact horizontal card. StatelessWidget — receives the request and an onTap callback. No state of its own.
lib/widgets/status_badge.dart
Colour-coded pill showing Open/Pending/Closed. Wrapped in a Hero widget. Open=green, Pending=amber, Closed=grey.

All Animations & Where They Are Implemented

Login → Dashboard
Type: FadeTransition · Duration: 400ms · File: NavigationService.fade() called from login_screen.dart · Why: Soft fade marks the transition from "outside" (login) to "inside" (the app). Slide would feel too abrupt here.
Any Screen → Request List / Detail
Type: SlideTransition (right) · Duration: 300ms · File: NavigationService.slideFromRight() · Why: Standard iOS/Android forward navigation convention. Feels natural to users.
FAB / Edit → Request Form
Type: SlideTransition (bottom) · Duration: 300ms · File: NavigationService.slideFromBottom() · Why: Rising from the bottom signals "this is a form/creation action" — distinct from regular navigation.
Logout → Login
Type: FadeTransition + pushAndRemoveUntil · File: Called from DashboardScreen logout action · Why: Stack is cleared so Back button doesn't return into the app. Fade is softer than a hard cut.
Dashboard Stat Cards on Load
Type: AnimatedOpacity + AnimatedSlide (staggered) · Duration: 500ms per card, 150ms stagger · File: dashboard_screen.dart initState() using Future.delayed · Why: Makes the dashboard feel alive. Draws the user's eye to the key numbers.
Image Picker Area (empty)
Type: AnimatedOpacity (repeat, opacity 0.5 ↔ 1.0) · File: request_form_screen.dart AnimationController · Why: Draws attention to an optional but important feature (photo upload). Stops pulsing once an image is selected.
Status Badge — Card to Detail Screen
Type: Hero animation · Tag: 'status-[request.id]' · Files: request_card.dart + detail_screen.dart · Why: Creates visual continuity between the list and the detail view — the badge physically "flies" from the card to the top banner.
Login Form Fields on Page Load
Type: AnimatedOpacity (mount fade-in) · File: login_screen.dart initState() · Why: The right panel appearing from opacity 0 → 1 gives the login screen a polished, premium feel on first open.

Data Flow: Creating a New Request

1. User taps FAB (+) on Dashboard or RequestListScreen
2. NavigationService.slideFromBottom() pushes RequestFormScreen (create mode)
3. Form auto-fills outletId, outletName, managerName from AuthService().currentUser
4. User fills in productName, productId, category, address, description
5. User optionally picks an image → XFile.path stored in state
6. User taps "Submit Request"
7. Form validates all fields → any error highlights that field red
8. New MaintenanceRequest created: id auto-generated (REQ-00X), date = today
9. Added to sampleRequests list via sampleRequests.add(newRequest)
10. SnackBar "Request submitted!"
11. Navigator.pop(context, newRequest) returns to calling screen
12. Calling screen receives the result, calls setState() to add card to list

Data Flow: Updating a Status

1. Manager taps [Update Status] button on DetailScreen
2. ModalBottomSheet shows Open / Pending / Closed options
3. Manager taps new status (e.g. Pending)
4. copyWith(status:'Pending') creates a new request object
5. statusHistory updated: "2026-07-06 14:30 · Open → Pending" appended
6. sampleRequests: old record replaced with new object (found by id)
7. setState() refreshes the detail screen UI
8. SnackBar "Status updated to Pending"
9. ModalBottomSheet closes
10. When user presses Back, Navigator.pop(context, updatedRequest) is called
11. RequestListScreen receives the updated object, updates the card in the list
